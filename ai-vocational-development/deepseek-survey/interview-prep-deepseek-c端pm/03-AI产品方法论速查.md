## 一、PMF（Product-Market Fit）

### 1.1 30 秒电梯
> "PMF = 你的产品在一个明确市场里满足明确需求，达到 retention + 自然增长拐点。Sean Ellis 的 '40% 用户说非常失望如果不能用' 是经典 quant 测试。"

### 1.2 关键测量
| 信号 | 阈值 |
| --- | --- |
| Sean Ellis test | >40% 用户 "very disappointed if removed" |
| Week-N 留存 | 稳态 Week-4 retention >25-30% (B2C) |
| 自然增长 | >20% 来源 word-of-mouth |
| NPS | >50 |
| 使用频次 | 与 hypothesis 一致（如 ChatGPT 期望 daily, Photoroom 期望 weekly）|

### 1.3 AI 产品 PMF 特殊性
1. **能力涌现** = PMF 拐点：模型升级可能突然让没 PMF 的产品达到 PMF
2. **场景定义模糊**：通用 LLM 不像传统 SaaS 有清晰 vertical
3. **用户期望膨胀快**：今天 PMF 明天可能因 GPT-5 不 PMF
4. **Wow → 流失漏斗**：尝鲜爆但留存难（"用过就走"）

### 1.4 AI PMF 检测的 5 个 specific signal
1. 用户**自发**带去新场景（不是产品教育的）
2. 高频回访（>3 次/周 for chat 类）
3. 用户帮你做 marketing（截图分享）
4. 付费意愿（即使产品 ugly）
5. 用户开始抱怨（"为什么你们 X 这么慢"——in love 才会抱怨）

### 1.5 高分话术
> "AI 产品 PMF 不是一次到位，是阶梯式——每次模型升级是一次重置机会。DeepSeek R1 的 reasoning 突破让 '深度思考' 这个 niche 突然有了 PMF；下一次 V4 可能解锁新 niche。PM 工作就是**预判 PMF 拐点 + 提前 ship 抓住**。"

## 二、用户增长（Growth）

### 2.1 AARRR / RARRA 漏斗

```
传统 AARRR：
  Acquisition → Activation → Retention → Revenue → Referral

现代 RARRA（强调留存）：
  Retention → Activation → Referral → Revenue → Acquisition
```

> **AI 产品建议用 RARRA**——AI 产品天然有"尝鲜流量"，没留存就是漏水桶。

### 2.2 增长循环（Growth Loop > Growth Funnel）

```
Funnel：用户进来 → 转化 → 流失（一次性）
Loop：用户行为 → 产生输出 → 输出带来新用户 → 闭环

经典 Loop 例子：
- ChatGPT：用户对话 → 分享链接 → 新人看到 → 注册
- Perplexity：用户 query → URL 分享 → SEO → 新用户
- 豆包：用户创建智能体 → 智能体被发现 → 新用户
```

### 2.3 AI Chat 的核心 Growth Loop 3 种

| Loop | 怎么 trigger |
| --- | --- |
| **分享 loop** | "分享对话" 按钮 + SEO 友好链接 + Twitter card 优化 |
| **UGC loop** | Custom GPT / Artifact / Project share → 公开广场 |
| **嵌入 loop** | Embed 代码 / Widget → 第三方 site 引流回 |

### 2.4 渠道矩阵（AI Chat 经典）

| 渠道 | DeepSeek 当前 | 成本 | 注 |
| --- | --- | --- | --- |
| 自然搜索 / SEO | 强（品牌词）| 0 | 优势渠道 |
| 微信生态 | 弱 | 中 | 可深挖（被元宝拦截）|
| 应用商店 ASO | 中 | 低 | 长尾词优化 |
| 媒体 PR | 强（每次模型发版）| 0 | DeepSeek 强项 |
| 投放（广告）| **不做** | 高 | 文化不匹配 |
| Influencer / KOL | 部分 | 中 | 海外有效 |
| 学校 / 社区 | 弱 | 低 | 学生场景潜力大 |
| 集成 / API 反哺 | 强 | 0 | 通过 API 用户认识 brand |

### 2.5 高分话术
> "DeepSeek 不投流，所以 growth 全靠 product-led + brand-led。前者要打磨核心体验做 word-of-mouth；后者要持续发布让媒体说话。这 2 个 lever 都要 PM 主动 design——分享按钮 / SEO 优化 / 每季度 hero feature ship。"

## 三、留存（Retention）

### 3.1 留存曲线 3 大形态

```
A) 微笑曲线（理想）：DAU 跌后回升，达到稳态
   → 真 PMF；用户回来

B) 滑梯曲线（差）：持续下跌不止
   → 无 PMF；产品 leaky bucket

C) 平台曲线（中）：跌后稳在某低点
   → 有部分 PMF；找 retention 触发点
```

### 3.2 留存指标体系

| 指标 | 公式 | 注 |
| --- | --- | --- |
| Day-1 | Day-1 活跃 / Day-0 注册 | onboarding 强弱 |
| Day-7 | Day-7 活跃 / Day-0 注册 | habit 形成 |
| Day-30 | Day-30 活跃 / Day-0 注册 | long-term PMF |
| W-N retention | Week N 活跃 / Week 0 | 中长期 |
| L-N (Last N days active) | 每 30 天活跃天数 | 深度指标 |
| Quick ratio | (新增 + 回流) / 流失 | growth health |

### 3.3 AI Chat 留存的核心 4 levers

| Lever | 例 |
| --- | --- |
| **Aha moment 复现** | 首次 wow → 第二天让用户能再次 wow |
| **Habit hook** | 早晨问候 / 每日推荐场景 |
| **Context 持久化** | Claude Projects / GPT memory |
| **场景拓展** | 让用户从 "我只用它写邮件" 扩到 "我用它做 N 件事" |

### 3.4 留存 debug framework

```
留存掉了？拆 4 维度：
1. 时间维度：什么时候开始掉？版本变更 / 算法变更 / 季节
2. 用户维度：哪类用户在掉？新 vs 老 / iOS vs Android / 地域
3. 行为维度：哪类行为在掉？深度对话 / 长 chat / 特定场景
4. 渠道维度：哪个 acquisition channel 用户在掉？quality issue
```

### 3.5 高分话术
> "留存第一性是 'aha moment 复现频率'——用户第一次用很 wow，第二次还能 wow 吗？DeepSeek R1 的 reasoning 第一次 wow，但如果第二次用户依然只问简单问题，wow 不复现，留存就掉。我会做：(1) 每日推送一个 reasoning 场景 trigger；(2) 历史对话 highlight 让用户回看；(3) UGC—分享好对话 + 别人看到学习。"

## 四、AI UX 设计原则（**C 端 PM 必背**）

### 4.1 5 大核心原则

#### 1·Show, Don't Tell（让 AI 能力可见）
- ❌ "我们模型很强"
- ✅ 让用户看到 thinking / reasoning trace / tool call

#### 2·Progressive Disclosure（渐进展示）
- 默认简洁
- 用户需要 detail 时展开
- 不要一上来用 50 个按钮吓人

#### 3·Honest about Limitations（透明边界）
- 不知道 → 说不知道
- 幻觉 → 标注 confidence
- 慢 → 提前告知

#### 4·Steering, Not Wand（用户控制 > 自动魔法）
- 让用户能 edit / regenerate / fork
- 不要 AI 单边决定一切
- Stop button 必须 prominent

#### 5·Trust through Transparency（用透明换信任）
- 显示 source citation
- 显示 thinking process
- 显示 model used

### 4.2 AI Chat 的 UX 雷区

| 雷区 | 例 | 修正 |
| --- | --- | --- |
| **No stop** | 用户想中断没法 | prominent stop button |
| **No regenerate** | 不满意没法重来 | regenerate / variants |
| **Thinking 不可见** | reasoning 5 秒空白用户以为坏 | streaming + progress |
| **Context blackbox** | 用户不知什么被 remember | 显示 context window / memory state |
| **Hallucination 不标** | 用户当真信 | citation / confidence / disclaimer |
| **Loading 不友好** | spinner 转 30 秒 | progressive content / skeleton |

### 4.3 AI Chat "好 UX" 例（每个抄一个 idea）

| 产品 | 好 UX | DeepSeek 可学 |
| --- | --- | --- |
| Claude Artifacts | 右侧 panel render code/HTML | Reasoning trace 可以右侧 panel |
| ChatGPT Voice | 全屏交互 / 自然语调 | 加 voice mode |
| Perplexity Citation | 每段后 footnote citation | 强化搜索 citation |
| Notion AI | Inline edit / 内嵌 | 集成可编辑文档 |
| Kimi 探索版 | "5 步骤"形式展示 reasoning | 借鉴 reasoning 可视化 |

### 4.4 高分话术
> "AI UX 的核心是 '让模型能力被感知 + 让用户有 control'。DeepSeek R1 的 thinking 是 hidden 在折叠里——用户不知道在 think，体验上模型 'didn't do anything for 10 seconds'。如果改成左侧 chat / 右侧实时 thinking timeline (像 cursor 那样)，用户 perceived value 直接翻倍。"

## 五、数据飞轮（Data Flywheel）

### 5.1 概念
> "用户使用 → 产生数据 → 反哺训练 → 模型变强 → 用户更满意 → 更多使用——AI 产品的 compounding moat。"

### 5.2 经典 AI 数据飞轮 3 种

| 飞轮 | 例 |
| --- | --- |
| **Preference feedback** | 用户 thumbs up/down → 训 reward model → RL → 模型更对齐 |
| **Failure mining** | 用户 regenerate / 抱怨 → 找 fail trajectory → SFT 补齐 |
| **场景探索** | 用户尝试新场景 → mine 高价值场景 → 训练专门提升 |

### 5.3 C 端 PM 在飞轮里的角色

```
1. 设计反馈机制（UX 友好的 thumbs up/down / 简短理由）
2. 定义 fail / success 信号（不是只看 explicit 评分）
3. 和算法 / 研究员对齐数据 spec
4. 平衡 "收集数据" vs "用户隐私"
5. 闭环 monitor —— 模型迭代后 user-perceived 变化是什么
```

### 5.4 隐藏信号（不靠 explicit thumbs）

| 信号 | 含义 |
| --- | --- |
| Regenerate | 用户不满意当前回答 |
| Quick session end | 回答没解决问题 |
| Long session + 高对话深度 | engagement 高 |
| Copy-paste 频率 | 用户在用回答 (positive) |
| 修改后 resubmit | clarify need |
| Negative tone in follow-up | 用户在抱怨 |

### 5.5 高分话术
> "C 端 PM 设计反馈机制最大坑是 '只关心 explicit 信号'——thumbs down 率 <2%，data sparse。真正大金矿是 implicit 信号：regenerate / 短 session / copy 率。我会和算法约定 5 个 implicit signal schema，自动 mine 月级别 10K+ 'silent dissatisfaction' trajectory 给 RM 训练。"

## 六、AI 能力边界（**C 端 PM 必懂**）

### 6.1 LLM 当前能 / 不能（2026 视角）

#### LLM **能**：
- 通用对话 / Q&A
- 写作 / 翻译 / summarization
- Coding（中小型 task）
- Reasoning（数学 / logic，o1/R1 类）
- Function calling / Tool use
- 长 context 理解（200K+）
- 多模态理解（image, audio - some）
- Agent task（短 horizon）

#### LLM **不能 / 弱**：
- **绝对事实**：会幻觉，需外部 grounding
- **实时信息**：训练 cutoff
- **极长 horizon agent**（>50 步）：易跑偏
- **复杂数学/科研突破**：人类 frontier 数学
- **多步骤精确执行**：金融交易、医疗诊断不能 fully autonomous
- **个人记忆**：默认无 long-term memory
- **物理感知 / 实时感官**：无 embodiment

### 6.2 "AI 能力转化产品" framework

```
给一个新模型能力 X，问 4 个问题：
1. 这个能力解决什么 user pain？
2. Best UX 是什么？（Chat? Sidebar? Inline? Voice?）
3. 失败时 fallback 是什么？
4. 用户教育成本多大？
```

### 6.3 例：R1 reasoning 能力如何变 C 端功能

| 4 问 | 答 |
| --- | --- |
| Pain | 用户问难题想看到 "AI 是怎么思考的"；不只要答案 |
| UX | 默认折叠（不打扰）；难题自动展开 thinking |
| Fallback | reasoning 失败 → 退回 standard 模式 |
| 教育 | "深度思考" toggle + 第一次用 tooltip 教育 |

### 6.4 例：multimodal 能力如何变 C 端功能

| 4 问 | 答 |
| --- | --- |
| Pain | 拍照问 / 看图说话 / 解题 |
| UX | 摄像头一键进入；预览 + 标注；输入框拖图 |
| Fallback | image 不清晰 → 提示拍清 |
| 教育 | 拍照按钮 + 短引导 |

### 6.5 高分话术
> "C 端 PM 翻译模型能力最常犯错是 '能力驱动'——'我们有 reasoning 所以做 reasoning 模式'。正确逻辑是 '场景驱动 + 能力赋能'——找用户高频痛点，用模型能力解掉，UX 让用户 0 学习成本 access。R1 reasoning 不是要做 'reasoning 模式' 而是 '当用户问难题时，自动深度思考给最好答案'。能力是 invisible 的更高级。"

## 七、辅助 framework 集锦（必记 4 个）

### 7.1 JTBD（Jobs-To-Be-Done）
> "用户雇佣你的产品来做什么 job？" 重新 frame 需求。

### 7.2 ICE 优先级（Impact × Confidence × Ease）
> 每个 feature 给 1-10 分，total = I × C × E

### 7.3 HEART 框架（Google UX 指标）
- **H**appiness, **E**ngagement, **A**doption, **R**etention, **T**ask success

### 7.4 North Star Metric + Inputs
- 北极星 1 个
- 2 级 4-6 个（北极星的 input）
- Guardrail 2-3 个（不能因北极星牺牲的）
