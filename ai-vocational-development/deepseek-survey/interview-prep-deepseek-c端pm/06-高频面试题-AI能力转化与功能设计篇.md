# 06 · 高频面试题·AI 能力转化与功能设计篇

> JD 职责 2 + 4 直接命中：**"理解 AI 能力边界，将 AI 能力转化为可落地的产品功能"** + **"深度理解大模型技术"**。
>
> 这一份是 C 端 PM 区别于"传统 PM"的最大差异化题型——10 道题 + 答题模板。

---

## 一、答题"通用 4-step 模板"

```
1. CAPABILITY UNDERSTANDING (30 秒)
   - 这个 AI 能力本质是什么（用技术语言精准描述）
   - 它的 capability boundary（能 / 不能）

2. USER SCENARIO MAPPING (60 秒)
   - 哪类用户在哪个场景能从这个能力受益
   - 不止 1 个场景，列 2-3 个

3. FEATURE DESIGN (90 秒)
   - 具体功能形态（UX / 入口 / 默认行为）
   - 失败 fallback
   - 用户教育成本

4. RISK + METRIC (30 秒)
   - 最大 risk
   - success metric
   - A/B 设计
```

## 二、10 道高频题

### Q1·"R1 reasoning 模型怎么设计成 C 端用户可感知的功能？"

**示范答**：

> **CAPABILITY**：R1 是 reasoning 模型——模型在回答前会自己 thinking 几千到几万 token (CoT)；本质是用 test-time compute 换 accuracy。Capability boundary：复杂数学 / 多步骤 logic 强；简单 query 用 reasoning 是 overkill；幻觉仍存在但 'systematic error' 减少。
>
> **SCENARIO**：3 类场景受益：
> - **学生场景**：解数学 / 物理 / 化学题
> - **知识工作者**：复杂决策 / 多步骤分析（financial modeling）
> - **专业 query**：法律 / 医疗 / 代码 debugging
>
> **FEATURE**：3 个层次设计：
>
> **Layer 1·Smart Auto-Routing（核心）**：
> - 默认 reasoning OFF
> - 小 classifier（50M 参数）判断 query 是否 benefit from reasoning
> - YES → 自动开 reasoning + UI 显示 "深度思考中..."
> - 用户可手动 toggle override
>
> **Layer 2·Thinking Stream UI**：
> - Reasoning 过程实时 streaming 显示（不是 spinner 8s）
> - 移动端折叠简版 ("正在思考第 N 步")
> - Web 端可展开看 full trace
>
> **Layer 3·Reasoning Insight 沉淀**：
> - 复杂 reasoning 完成后可 'pin to project'
> - 历史可回看 + 分享（reasoning trace 也是有价值内容）
>
> **RISK**：
> - Auto-routing 误判 → 简单题等 8s 用户烦
> - Mitigation：classifier confidence threshold + 用户主动 toggle 永远可用
>
> **METRIC**：
> - 北极星：reasoning 模式贡献的'问题解决率'（A/B vs 标准模式）
> - 副：reasoning 模式覆盖率、Day-7 留存差
> - A/B 设计：N=50K，主指标问题解决率（用户 self-report + thumbs up），期 effect +5pp，可达 95% 显著

### Q2·"如果 DeepSeek 加 voice 功能，你怎么设计？"

**示范答**：

> **CAPABILITY**：Voice = TTS + STT + voice activity detection + 可选模型 voice-to-voice (如 GPT-4o realtime)。当前 DeepSeek 没自研 voice model，需要 (a) 集成第三方 (MiniMax / 字节) 或 (b) 自研。
>
> **SCENARIO**：3 类场景：
> 1. **通勤 / 开车**：手不方便用键盘
> 2. **快速 brainstorm**：思路快不想打字
> 3. **学语言**：voice 对话练口语
>
> **FEATURE**：分 3 阶段 ship：
>
> **V1·Phase 1 (1 月)·Push-to-Talk**：
> - App 加 voice 录音按钮
> - 录音 → STT → 当文本送 LLM → TTS 播放回答
> - 简单实现但用户能 feel
>
> **V2·Phase 2 (3 月)·Continuous voice mode**：
> - 全屏 voice UI（仿 ChatGPT Voice）
> - VAD 自动检测说话停 → 发送
> - 模型回答可被打断
>
> **V3·Phase 3 (6 月+)·真 voice-to-voice**：
> - 集成 native multimodal model（自研或合作）
> - 真低延迟（< 800ms）
> - 情感识别 / 多 voice 选择
>
> **RISK**：
> - **质量**：voice quality 不如豆包 → 早期用户失望
>   - Mitigation：先 limit beta 给 power user，质量 ready 再 GA
> - **成本**：voice 用量大 → cost 爆
>   - Mitigation：voice 模式可能优先付费版
> - **隐私**：录音敏感
>   - Mitigation：本地 VAD + 不存储 raw audio
>
> **METRIC**：
> - Voice 模式 DAU 占比（目标 6 月内 15%+）
> - 移动端 Day-7 留存 vs 无 voice 用户（期 +8-12pp）
> - Voice session 平均时长 vs text session（期 voice 比 text 长 1.5x）

### Q3·"假如模型新增 long context (1M tokens)，C 端怎么用？"

**示范答**：

> **CAPABILITY**：1M context = 一次能塞 ~750K 字 (一整本书 / 100 篇 paper / 10 万行代码)。Capability boundary：long context 仍有 'lost in the middle' 问题；attention 衰减；cost 飙升。
>
> **SCENARIO**：4 类场景：
> 1. **学生 / 研究**：上传 paper / 教材 + 问答
> 2. **法律 / 合同**：长合同审阅 + 风险识别
> 3. **代码 review**：整个 repo 上传 + 分析
> 4. **历史对话**：所有过去对话作 context（personal memory）
>
> **FEATURE**：
>
> **Feature 1·Document Upload（基础）**：
> - 支持 PDF / DOCX / TXT 上传 (单文件 50MB)
> - UI 显示 'X% of context window used'
> - 用户可见 + 可控
>
> **Feature 2·Projects / Workspace（高级）**：
> - 类似 Claude Projects
> - 用户建 'Project'，所有 doc 持久存
> - 跨对话共享 context
> - 重度用户 retention 神器
>
> **Feature 3·Memory / Personal Context（个性化）**：
> - 用户可勾选 '记住这个事实'（如 'I'm a backend engineer')
> - 跨对话自动 inject
> - 用户可见 / 可删
>
> **RISK**：
> - **Cost**：1M context query 比 short 贵 100x → 必须有 budget control
>   - Mitigation：先 free user 限 100K，付费 1M
> - **Quality**：Long context 模型可能 'lost in middle'
>   - Mitigation：用 RAG retrieval 而非全 context 注入
> - **隐私**：上传 doc 敏感
>   - Mitigation：明确数据处理 policy + 用户可删
>
> **METRIC**：
> - Document upload 用户的 Day-30 留存（期 vs 不上传用户 +15pp）
> - Projects 用户的 weekly 活跃天数（期 5+）
> - Long context query 占比（期 6 月内 10%+）

### Q4·"幻觉怎么 product-level 缓解？"

**示范答**：

> **CAPABILITY 真相**：幻觉是 LLM 本质——模型不知 'I don't know'，会 confident 产生不存在的事实。完全消除不可能；只能 reduce + manage。
>
> **5 个 product-level lever**（按 ROI 排）：
>
> **Lever 1·Grounding to retrieval（最有效）**：
> - 联网搜索 + 引用外部 source
> - 每段答案附 citation（Perplexity 风格）
> - 用户能 verify
> - 效果：可信度 +50%+；hallucination 感知 -70%
>
> **Lever 2·Confidence 显示**：
> - 模型对答案有 confidence score (通过 perplexity / self-evaluation)
> - 低 confidence 标 '⚠️ 我不完全确定，建议核实'
> - 教育用户'AI 不是 100% 对'
>
> **Lever 3·Domain-specific guardrail**：
> - 医疗 / 法律 / 金融 等高 risk domain
> - 强制显示 disclaimer + 推荐咨询专业人士
> - 部分 query 拒答 + 引导
>
> **Lever 4·User feedback loop**：
> - 用户能 thumbs down + 标 'hallucination'
> - 数据反哺训练（避免类似 fail）
> - 显示 'X% users found this helpful'
>
> **Lever 5·Reasoning 模式 surface**：
> - Reasoning 模型 hallucination 显著降低
> - 在 high-stakes query 自动开 reasoning
>
> **优先级**：Lever 1 + 2 必做（产品层面）；Lever 3 行业相关；Lever 4-5 中长期。
>
> **不该做**：
> ❌ 在所有回答前都加 disclaimer——用户脱敏失去信任感
> ❌ 拒答太多——用户跑去竞品
>
> **关键 mindset**：幻觉是不可消除的 LLM 本质，PM 的工作是让 'user expectation calibrated to model capability'——不是骗用户 AI 不会错，是教用户怎么 use AI critically。

### Q5·"如果模型 latency 从 1s 涨到 5s，PM 怎么应对？"

**示范答**：

> **场景**：模型质量升级（如 R1 → R1.5）但 latency 增加——质量 vs 速度 trade-off。
>
> **第 1 步·量化 impact**：
> - Latency 增加对 'completion rate' / 'session length' 影响 (A/B test)
> - 经验：每 +1s latency，conversion -5~10%
> - 估算用户流失数 vs 质量提升带来的 retention 增益
>
> **第 2 步·分层应对**：
>
> **Layer A·Engineering (找算法 / infra)**：
> - 能否 keep 质量但优化 inference？(quantization, speculative decoding, KV cache)
> - 能否 dual-model: 简单 query 用快 model，复杂 query 用慢 high-quality model
>
> **Layer B·Product UX**：
> - **Streaming 优先**：首字延迟 < 1s，整体 5s 用户感知 OK
> - **Progressive content**: 先展示 skeleton / outline，再 fill 内容
> - **Thinking visualization**: latency 用来展示 'thinking process'，把 wait 变 value
> - **Background generation**: 用户继续输入下一个 query，前一个在后台
>
> **Layer C·Smart routing**：
> - 简单 query 路由到快 model
> - 复杂 query 才上慢但好的 model
> - 用户大部分 query 仍快
>
> **第 3 步·测量 + 决策**：
> - 设 A/B: 新慢 model vs 旧快 model
> - 主指标：用户满意度 (thumbs up + retention)
> - 副：latency, completion rate
> - 决策矩阵：
>   - 满意度↑ & retention 不降 → ship 新 model
>   - 满意度↑ but retention↓ → 看 net effect，可能需 routing
>   - 满意度不变 → keep 旧 model
>
> **特别注意**：
> 1. **Latency 是感知性的，不是绝对的**——5s 显示 thinking 比 5s 空白 spinner 体验天差地别
> 2. **不同场景容忍度不同**——reasoning 模式用户接受 30s，闲聊用户不接受 3s
> 3. **不要因 latency 牺牲质量**——但要把 latency 时间 'productize'

### Q6·"DeepSeek 应不应该加 image generation？"

**示范答**：

> **决策框架·5 个 question**：
>
> **Q1·用户真的要吗？**
> - 当前 DeepSeek 用户偏理性（reasoning / writing），image gen 需求 vs 通用 chat 用户低
> - 但 user research 应 surface，不 assume
>
> **Q2·我们能做好吗？**
> - DeepSeek 没有自研 image model（Janus 是 understanding，不是 strong generation）
> - 自研需 6-12 月 + 大量 compute
> - 替代：集成第三方（Doubao / Midjourney / SD）
>
> **Q3·有多大 strategic value？**
> - 完整度提升（让 DeepSeek "all-in-one")
> - 但 cost：infra cost + 团队 focus 分散
> - 边际：豆包 / ChatGPT image 已强，DeepSeek 追赶难差异化
>
> **Q4·机会成本是什么？**
> - 同样 compute 投到 reasoning V4 / coding agent 可能 ROI 更高
> - PM bandwidth 投这边 = 其他不做
>
> **Q5·竞争压力多大？**
> - 用户因为没 image 离开 DeepSeek 的比例？我估 <5%
> - vs ChatGPT 有 image 但很多用户仍用 DeepSeek for reasoning
>
> **我的建议**：
> - **短期不做**——focus on reasoning + writing + code (DeepSeek 强项)
> - **中期 V1**：集成第三方 (Doubao 或开源 SD)——cost 低 + 满足 basic 需求
> - **长期 V2**：等模型团队 ready 再自研——如果 multimodal 是公司 frontier 战略
>
> **如果决定做，最小化 product 设计**：
> - 入口在对话框内，不单独 tab
> - 提示 '由 X 模型生成'（透明）
> - 不投入 image editing / inpaint 等深度功能
>
> **关键 mindset**：**not building 也是 product decision**——好 PM 知道说 no，把 bandwidth 留给 high-ROI bet。

### Q7·"如果模型 emerge 一种新 capability（如 model can plan 100-step task），怎么 productize?"

**示范答**：

> **第 1 步·真验证 capability**：
> - 模型团队 claim ≠ 用户实际可用
> - 自己 dogfood 30 个真实 long-horizon task：成功率 / 失败模式
> - 量化：从 baseline 5% 提到 30%? 50%? 80%?
>
> **第 2 步·找 high-value scenario**：
> - 哪些 long-horizon task 用户真有？
>   - Code refactor 整个 module
>   - 写一本 e-book draft
>   - 旅行规划 + 订票
>   - 学习计划 + 跟进
> - 优先 verifiable scenario（task 是否完成可判断）
>
> **第 3 步·MVP feature 设计**：
> 
> **MVP·"Agent Mode"**：
> - 入口：对话内 toggle "agent mode"
> - 用户给 high-level goal
> - 模型自动 plan + 执行（可能调 tool / 写代码 / 访问 web）
> - 实时显示 plan + progress + 每步结果
> - 用户可中途介入 / 修正 / 终止
>
> **UX 关键**：
> - **Plan transparency**：用户看到 "next step: X, then Y"
> - **Checkpoint approval**：高 risk 步骤（如 send email / pay）必 user confirm
> - **Graceful fallback**：执行失败时清晰 error + retry / skip 选项
>
> **第 4 步·分阶段 rollout**：
> - Phase 1·beta：限定 power user，受限 task type
> - Phase 2·限定 GA：成功率 >70% 的 task type 默认
> - Phase 3·full GA：unrestricted
>
> **RISK**：
> - **Trust erosion**：失败 task 让用户 lose faith
>   - Mitigation：超严过滤，只发布成功率高的 task type
> - **Cost spike**：long task 10x token
>   - Mitigation：付费版独享 / 免费版限次
> - **Safety**：autonomous action 可能犯 high-stake error
>   - Mitigation：checkpoint approval + audit log
>
> **METRIC**：
> - Agent task 成功率（期 60%+ 才 GA）
> - Agent 用户的留存（期 +30pp vs 非 agent 用户）
> - 用户 cancel agent 率（期 <20%）

### Q8·"Memory 功能（跨对话记住事实）该不该加？"

**示范答**：

> **YES，但要谨慎设计**。3 个考虑维度：
>
> **Value**：高
> - ChatGPT memory 上线后 retention 显著提升
> - 重度用户的 'cumulative value' 飞轮
> - 个性化体验直接提升
>
> **Risk**：高
> - **隐私**：用户敏感信息被存
> - **错误记忆**：模型错 inject fact 持续影响
> - **GDPR/合规**：用户有 right to be forgotten
>
> **设计原则**：
> 1. **Opt-in**——default OFF，用户主动开
> 2. **Transparent**——用户可查看所有 memory items
> 3. **Editable**——逐条 delete / edit
> 4. **Controllable**——可暂停 memory 当前对话
> 5. **Sensitive filter**——不存 financial / health 等高敏感
>
> **UX 形态**：
> - Memory 入口在 settings + 对话内 footer
> - 对话中 'memory updated: X' inline 提示（不打扰但可见）
> - Memory page: 可删 / 可 search
>
> **Phase rollout**：
> - Phase 1·beta：100K 用户，收集 trust / privacy feedback
> - Phase 2·opt-in GA
> - Phase 3·可考虑 default ON for new users（需 PR 和合规准备）
>
> **METRIC**：
> - Memory 开启率（期 30%+）
> - Memory 开启用户的 Day-30 留存（期 +15pp）
> - Memory delete 率（如 >40% 说明误判太多）

### Q9·"如果 tool use / function calling 在 C 端做，第一个 tool 选什么？"

**示范答**：

> **选择 framework·4 criteria**：
> 1. **High user value**：覆盖大用户 group + 高频
> 2. **Low integration cost**：API stable + 公开
> 3. **Verifiable result**：调成功/失败可自动判断
> 4. **Safe**：不会造成实质 harm
>
> **Top 候选 + 评估**：
>
> | Tool | Value | Cost | Verifiable | Safe |
> | --- | --- | --- | --- | --- |
> | **Web search** | ✅✅ 高频 | ✅ 已实现 | ✅ | ✅ |
> | **Code execution** | ✅✅ 数学/编程 | ✅ sandbox | ✅✅ | ⚠️ 需 sandbox |
> | **File reading** | ✅ | ✅ | ✅ | ✅ |
> | **Image generation** | ✅ | ⚠️ 第三方依赖 | ✅ | ✅ |
> | **Email send** | ⚠️ | ⚠️ OAuth | ⚠️ | ❌ 高 risk |
> | **Calendar** | ⚠️ | ⚠️ OAuth | ⚠️ | ⚠️ |
> | **Shopping (订购)** | ❌ | ❌ | ⚠️ | ❌ 高 risk |
>
> **第一个 tool 推荐：Code execution (Python sandbox)**
>
> **理由**：
> - 数学 / 编程是 DeepSeek 强场景，code exec 直接放大
> - 完全 verifiable（运行成功 = success）
> - Sandbox 内安全
> - 复刻 ChatGPT Code Interpreter 经验丰富
>
> **次选**：Web search（虽然已有，但可深化——Pro Search / Multi-step）
>
> **要避开**：
> - Email / Calendar 等需 OAuth 高敏感 tool
> - Shopping / payment 等可能 financial loss tool
> - 需要 user account binding 的复杂 tool
>
> **UX 设计**：
> - 用户问 '解这个方程' → 模型自动 'run python' 显示代码 + 结果
> - 完全 transparent：代码 + 输出都展示
> - 用户可看 / 修改 / 重跑
>
> **METRIC**：
> - Code exec 触发率（数学/code query 中 >50%）
> - 包含 code exec 的回答用户满意度 vs 不包含（期 +20pp）

### Q10·"AI 产品最常见的 anti-pattern 是什么？"

**Philosophical reflective question** —— 看你的 product wisdom。

**示范答**：

> "我观察到 5 个最常见 anti-pattern（按严重排）：
>
> **#1·Capability worship (能力崇拜)**：
> 模型新能力出来 → '我们快加'，没问 'user 要吗'。常见症状：硬塞 reasoning / agent mode 到所有 query；多模态当噱头无实际价值。
> **解药**：JTBD framework——先问 'user jobs'，再问 'capability fit'。
>
> **#2·Hype-driven roadmap**：
> Twitter / 朋友圈最近 trend 什么就追什么。常见症状：今年 agent 火→all in agent；明年长 context 火→所有功能都 1M context。
> **解药**：Hype cycle 意识——成熟期再 commit，trough 期沉淀。
>
> **#3·Chat-only-ism**：
> 假设所有 AI 交互都是 chat。常见症状：明明用户 jobs 是 'edit doc'，UX 强行 chat 形态。
> **解药**：考虑 Canvas / Sidebar / Inline / Voice / Background 等多种 modality。
>
> **#4·Magic over Control**：
> AI 单方面决定一切，用户被动。常见症状：模型自动改用户文件无 confirm；agent 自主行动无 checkpoint。
> **解药**：'steering wheel, not autopilot'——总让用户保留 control。
>
> **#5·Metric tunnel vision**：
> 盯短期 metric ↑，忽视 long-term 危害。常见症状：用红包推 DAU；用 'addictive' 设计推 engagement (但用户后悔)。
> **解药**：guardrail metric (Day-30 留存 / NPS) + 'after-use feeling' qualitative。
>
> **对 DeepSeek**：
> - #1 capability worship 是我最担心 risk—— '我们 R1 这么强，必须把 reasoning 塞到所有地方'。但用户大部分不需要 reasoning。我会 advocate 'user-first, capability-empowered' 路线。
> - #4 magic over control 反而 DeepSeek 还好——开源 + 透明文化天然防御。
>
> 健康 anti-pattern 防御 = 写下来 + team 内部 review feature 时 check list。"

## 三、面试官最爱追问的 5 个角度

1. **"你会怎么 validate 这个 hypothesis？"** → A/B test 设计
2. **"如果只能做 1 件，做哪个？"** → 测 priority
3. **"用户教育成本你怎么 estimate？"** → 看你的 user empathy
4. **"模型不 capable 怎么办？"** → fallback 设计
5. **"长期看这个 feature 5 年后还存在吗？"** → strategic thinking

## 四、最后忠告

> **AI 能力转化题 = 这个岗位的 differentiator。**
