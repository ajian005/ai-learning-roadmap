# 06｜高频面试题 — 数据/训练/评测闭环 + 实验设计篇

> 数据是 DeepSeek 这个岗位"职责 2"——和数据标注团队协作设计标注方案、找模型短板、闭环改进。

> 实验设计是研究员的"硬功夫"——能不能正确 design + 解读实验是 senior 和 junior 的分水岭。

> 共 12 题。

---

## Q1. 如何设计一份高质量标注 guideline？

### 满分骨架（SOP）

```
Step 1: 任务定义 (1 页)
- 一句话目标
- 5 个正例 + 5 个反例（必须）
- 边界 case：5~10 个易混淆的

Step 2: 标注 schema
- 输入字段 / 输出字段
- 每个字段 enum / 文本 / score 等
- 是否多选 / 是否可空

Step 3: 决策树 (flowchart)
- 遇到 X 怎么办
- 遇到 Y 怎么办
- 模糊情况怎么处理

Step 4: 质量标准
- gold set 100 题：标注员必须先过
- inter-annotator agreement: Cohen's kappa > 0.7（多分类）/ 0.85（二分类）
- 每周抽检 5%：错误率 < 5%

Step 5: 培训 + 校准
- 第一轮：标 50 题 → 评分 → 反馈
- 第二轮：标 100 题 → kappa 测算
- 通过率 < 80% 的标注员淘汰

Step 6: 持续改进
- 每周一次"难点会议"：讨论模糊 case
- 更新 guideline → 重新校准

Step 7: 数据清洗
- 自动检查 schema 合规
- 双标 / 三标的 majority vote 处理
- 不一致 case 单独 review
```

### 加分

- 提到 "**guideline 是活的**——每周 v1.1, v1.2 迭代"
- 提到 "用 LLM 当 second annotator——同时给人和 LLM 标，对照"
- 提到 "**让标注员参与 model 评测**——他们最敏感 model 弱点"

## Q2. 给你 1000 条 fail case，怎么做 error analysis？

### 满分

```
Step 1: 看一眼 5~10 条 → 形成"感觉"
- 不要先分类，先 immerse 在数据里

Step 2: 设计分类轴 (axes)
- Axis 1: 失败类型（understanding / reasoning / format / hallucination / refusal / tool）
- Axis 2: 难度（easy / medium / hard）
- Axis 3: 任务类型（math / code / general / agent）

Step 3: 采样 100 条精标
- 不要全标 → 太贵
- 每个 axis × 30~50 条够看 pattern

Step 4: 聚类 + 可视化
- 用 embedding（如 BGE）聚类 1000 条
- 看每簇代表什么 → "啊原来 30% 失败都是 prompt 太长"
- 用 GPT-4 给每簇起名

Step 5: 找 top-3 失败模式
- 占比 + 严重度 + 可改难度
- 排序：高占比 + 易 fix 优先

Step 6: 提改进 hypothesis
- 失败模式 → root cause hypothesis → 数据/训练改进方案
- 每个 hypothesis 配 1 个实验设计

Step 7: 反向验证
- 训完看新 model 这个失败模式是否减少
- 同时看其他 metric 是否 regression
```

### 易踩
- 不抽样直接 manual 全看 → 时间黑洞
- 不用 GPT-4/embedding 自动聚类 → 凭感觉分类容易漏

### 加分
- 提到 "用 mistake similarity 找 root cause——多个 fail 在同一个 hidden cause（如训练数据缺某 domain）"
- 提到 "fail case → 反向构造 training data" 的闭环

## Q3. 设计一个 "数据-训练-评测" 闭环

### 满分（DeepSeek 风格）

```
┌──────────────────────────────────────────────┐
│  评测 (Eval)                                  │
│  - 跑多 benchmark + held-out                  │
│  - 误差分析: 找 top-K 失败模式               │
└────────────┬─────────────────────────────────┘
             │
             ▼
┌──────────────────────────────────────────────┐
│  数据 (Data)                                  │
│  - 针对失败模式: 构造 training data           │
│    - 人工标 (高质量 但贵)                     │
│    - 自动合成 (LLM 生成 + 过滤)              │
│    - 公开数据 (筛选 + 清洗)                   │
│  - 数据质量门:                                │
│    - 去重 / 去 contamination                  │
│    - quality score (LLM judge)                │
│    - difficulty score (model 当前是否能解)     │
└────────────┬─────────────────────────────────┘
             │
             ▼
┌──────────────────────────────────────────────┐
│  训练 (Train)                                 │
│  - SFT (新数据 + replay 老数据)               │
│  - DPO / GRPO (preference / verifier)        │
│  - 多 seed 训练 + checkpoint averaging       │
└────────────┬─────────────────────────────────┘
             │
             ▼
        回到 Eval...
        
       (每轮 cycle 1~2 周)
```

### 关键设计原则

1. **每轮 cycle 越短越好**——T-2 周以内为佳，否则 feedback loop 太慢
2. **每轮只改 1 个 hypothesis**——多变量同改无法归因
3. **eval 必须独立** ≠ 训练集——held-out + 时间 cutoff
4. **数据 quality 比 quantity 重要**——10K 精数据 > 100K 噪数据
5. **建立 "regression test"**：上一版的 strength 不能掉

### 高频追问

**Q：怎么避免 overfitting 到失败模式？**
A：同时跑 general benchmark（MMLU/GSM8K/HumanEval）作 regression guard——如果 down > 1%，hold 该次发布。

**Q：什么时候不应该改？**
A：当 fail 是 fundamental model capability 限制（如 long context > 100K），加 data 帮助小——应该考虑架构改动或 inference-time scaling。

## Q4. 如何设计一个 Ablation Study？

### 满分

```
Ablation 目的: 验证每个 component 的真实贡献

设计原则:
1. 控制变量: 一次只改一个
2. 同 seed × 多 run: 跑 3+ seed，看 mean ± std
3. 同 baseline: 所有 ablation 都从同一 baseline 加减
4. Eval 一致: 同 eval set + 同 metric

实例（DPO ablation）:
  Baseline: SFT
  + DPO              → +X% (vs baseline)
  + DPO + KL(β=0.1)  → +Y%
  + DPO + IPO loss   → +Z%
  + DPO + length norm → ...

报告:
- 表格: 每行一个 config，每列一个 metric
- 必须报 std (或 95% CI)
- 必须报相对 baseline 的 delta
- 必须报 compute cost (GPU-hour)

避免:
- 只跑 1 个 seed 就下结论
- 用同 eval set 反复调参（hidden overfit）
- 不报 negative result
```

### 加分

- 提到 **"应该有 negative ablation"**——证明某 component **真的需要**（去掉它 → 掉点）
- 提到 **Bonferroni correction**——多次比较时 p-value 要校正

## Q5. 如何 detect 模型训练时的 contamination？

### 满分

**Contamination 定义**：评测题（或近似题）出现在训练数据中 → eval 分数虚高。

**Detect 方法**：

1. **N-gram overlap**：算 eval prompt 在训练集的 13-gram 命中率
2. **Embedding similarity**：BGE/MTEB 检索训练集，相似度 > 0.9 标记
3. **Memorization probe**：让 model 续写 eval prompt 的开头，看能否一字不差 reproduce
4. **Cutoff date check**：eval 数据是否在模型 cutoff date 之后
5. **LLM judge**：让 GPT-4 比较"这个 eval 题和训练集 X 题是不是同一个"

**Mitigate**：
- 训练数据严格去重
- Eval 用 cutoff date 后构造
- 用 **私有 / 未公开** eval set（如 OpenAI 的 GPQA Diamond + private）
- Holdout 一部分公开 benchmark 不参与训练
- 跨 model lab 用同一 eval（容易看到 contamination signal）

### 高频追问

- "如果发现 eval 被 contaminate 了怎么办？" → 1) 清掉训练数据重训（贵）2) 改用未污染 benchmark 3) 至少在 paper 中报告

## Q6. 实验设计题：你想验证 "PRM 比 ORM 在 reasoning 上更好"，怎么 design？

### 满分

```
Hypothesis: PRM-trained reward 比 ORM-trained 在 reasoning task 上更准 + 训出的 model 更强

Setup:
- Base model: 同一个（如 Qwen-7B）
- Task: 同一 reasoning benchmark（GSM8K + MATH）
- Train data: 同 prompts，区别只在 reward signal 形式
  - ORM 组: 只用 final answer 0/1
  - PRM 组: 用 step-level reward (Math-Shepherd 方式生成)

Metric:
- Primary: pass@1 on held-out test
- Secondary: pass@N, reasoning length, intermediate correctness

Controls:
- Same RL alg (e.g., GRPO)
- Same hyper-params (lr, β, batch size)
- Same training steps
- 3 seeds each, report mean ± std

Ablation:
- ORM 组用 best-of-N（PRM rerank） vs PRM 组直接 sample → 区分"训练效果"vs"inference 效果"
- 改变 PRM 质量（高/中/低 quality）→ 看 PRM-quality x final-perf curve

Pitfalls 预防:
- PRM 数据 quality 是混淆变量 → 用 Math-Shepherd 控制
- ORM 可能数据不够多 → 保证 step-level reward 总信号 ≥ ORM
- Test set contamination → cutoff date + 私有 GSM8K-like
- Length bias → 报告 token-normalized metric

Expected outcome:
- 假设 PRM > ORM 5~10 个 percentage point on MATH
- 如果不显著 → 报告 negative + 分析 (PRM noise? credit assignment 实际不重要?)

Report:
- 表 + 折线 (training step vs perf)
- 分布 plot (per-task perf)
- Discussion: limitation + future
```

### 易踩
- 只跑一次 + 不报 std → 不可信
- 比较时控制不全（如 ORM 用 lr=5e-6, PRM 用 lr=1e-5 → 不公平）
- 没设计 negative result 的解释方案

## Q7. 你的 model 训完后 GSM8K 涨了 5%, 但 IFEval 掉了 3%, 怎么办？

### 满分

**这是经典的 capability vs alignment trade-off 问题。**

**第一步：诊断**
- IFEval 下降是因为什么？读 100 条 IFEval fail case
- 看是 instruction following（如忽略 "answer in JSON"）还是 helpfulness 下降
- 是否 model 学会"想很久但忘指令"（CoT 长度暴涨 → 末尾 instruction 被遗忘）

**第二步：归因**
- 训练时是否过度 push reasoning，导致 IF 数据被 dilute？
- KL penalty 是否太小？（policy 漂得太远）
- 训练 mix 比例：reasoning vs IF data 是否失衡？

**第三步：决策**

| 情况 | 决策 |
| --- | --- |
| IF 下降是 critical user-facing 能力 | Hold release，调 mix 重训 |
| IF 下降在长 CoT 场景，可加 system prompt 缓解 | Ship 但加 mitigation |
| IF benchmark 本身不代表 user value | 与 PM 沟通，看用户反馈是否真受影响 |

**第四步：实验改进**
- 加 IF data 到 mix，调比例
- 加 IFEval reward 到 multi-objective RL
- 用 SFT regularization 防 alignment tax
- 用 model souping / weight averaging：take 训前 model + 训后 model 平均

### 加分

- 不立刻"鸡血"调参——先做 error analysis 理解原因
- 知道 **"alignment tax 不一定不可避免"**——好的训练 recipe 能让两者都涨

## Q8. 假设 PM 跟你说 "我们要让模型在 X 任务上提升 10%"，你前 3 天怎么 spend？

### 满分

```
Day 1 (Morning): 理解任务
- 拿到 X 任务的 representative samples（30~50 条）
- 自己手动做 5 条 → 体会难点
- 跑当前 model 100 条 → 分类 fail mode（top-3）

Day 1 (Afternoon): 数据现状
- 检查训练数据 mix 里 X 类任务占比
- 查现有 benchmark 覆盖
- 找最相关的公开数据集

Day 2: Hypothesis 设计
- Top-3 fail mode 各对应一个改进方案 hypothesis:
  - H1: 数据不够多 → 加 N 条
  - H2: 数据质量差 → filter + 加 verifier
  - H3: 算法不对 → 换 RL 算法
- 每个 hypothesis 设计一个 quick experiment（< 6h)

Day 3: 优先级 + Roadmap
- 写 1 页 doc:
  - 现状 % + 目标 % 
  - Top-3 hypothesis + cost/benefit
  - 推荐的实验顺序 + 时间表
  - 失败的 fallback
- 和 PM/team lead align

后续:
- 按 plan 跑实验，每 3 天一次 sync
```

### 关键

- **不要 day 1 立刻调参** —— 先理解问题
- **不要承诺 10%** —— 给一个区间 + 风险评估
- **要有 fallback** —— 如果 H1~H3 都不 work，备选 H4

## Q9. 如何 detect 一份数据的 quality？

### 满分

**自动指标**：
1. **Schema 合规率**：JSON parsable / 字段全 / enum 有效
2. **Length distribution**：异常长/短 sample 标记
3. **Repetition rate**：重复 n-gram 比例
4. **Perplexity**：用 reference model 算 ppl，过高过低都可疑
5. **Embedding diversity**：cluster 看是否过于集中
6. **LLM judge score**：用 GPT-4 给 quality score（1~5）

**人工抽检**：
- Random 100 条 → 人工 binary label "high quality?"
- 用此结果 calibrate 自动指标的阈值

**Contamination check**：
- N-gram match against eval set
- Embedding similarity to eval prompts

**Difficulty check**：
- 跑当前 model 100 条 → 全对的太简单（没信号），全错的太难（信号也低）
- 中等难度（model 当前 30%~70% 准确）的最有价值

### 加分

- 提到 "**diversity 比 quality 重要**" —— 100K 同质数据不如 10K 多样数据
- 提到 "**Active learning 思想**——挑 model 不确定 / 失败的样本优先标"

## Q10. 你怎么决定一个实验 "失败" 和 "成功"？

### 满分

```
成功定义:
- Primary metric 超过 baseline 显著（p < 0.05 / 95% CI 不重叠）
- Secondary metrics 不显著 regress
- Cost 在可接受范围
- 可复现（多 seed 一致）

失败也分类:
- 完全失败: primary 持平甚至下降 → 拒绝 hypothesis
- 部分成功: primary 涨但 secondary 掉 → 需 trade-off 讨论
- 不确定: 信号 marginal → 加更多 run / 改 experiment design

"失败"的价值:
- Negative result 也是 result
- 写在 wiki / blog——后人可避坑
- 反思 hypothesis 是否真的不对，还是 experiment 本身有问题

避免:
- "结果不好 → 不报告"（学术作弊）
- "结果不好 → 改 metric 直到好"（p-hacking）
- "结果不好 → 多试几次直到好"（fishing）
```

### 加分

- 提到 "negative result 写 internal blog post——DeepSeek 这种 research-driven 公司应该 reward 这种 share"

## Q11. 数据标注阶段的"质量校准会议"怎么开？

### 满分（具体 SOP）

```
频率: 每周 1 次, 1 小时
参会: 标注 lead + 3~5 个标注员 + 1 个研究员 + 1 个 PM

议程:
1. 上周 inter-annotator agreement 数字 (5 min)
2. 抽检失败率 + 典型错误 case (10 min)
3. 模糊 case 讨论 (30 min)
   - 每人带 3 条"我也不知道怎么标"的 case
   - 讨论 → 投票 → guideline 增量
4. 新 case type 学习 (10 min)
   - PM/研究员 announce 新需求或新边界
5. Action items (5 min)
   - guideline v1.X 更新
   - 重新校准的 timeline

研究员角色:
- 不主导决策 → 标注员是 expert
- 提供"为什么这样标对训练有用"的 context
- 把模糊 case 反馈到 model error analysis
```

### 加分

- 提到 "**让标注员看到 model output**——很多 case 标注员看了 model 才知道关键 distinction"
- 提到 "**reward 高质量标注员**——他们标的数据 model gain 更大，可以 share 这一信号"

## Q12. （开放）如果你发现训练数据有系统性 bias（比如全是 male 程序员的 code），怎么处理？

### 满分

**承认问题严重性**：
- 系统性 bias 会被模型放大
- 用户群体 underrepresented 会受影响
- 长期是 brand / ethics 风险

**短期 mitigation**：
1. 补充少数群体 data
2. 用 LLM 合成 "diversified" 版本
3. 训练时加 fairness constraint / penalty
4. Inference 时加 system prompt 引导平衡

**长期 fix**：
1. 修数据采集 pipeline——上游就要多样
2. 建立 fairness eval set + 持续监控
3. 招标注员时也要 diversity
4. 透明 reporting——公开 model card 说明 bias

**关键 mindset**：
- 不 defensiveness——承认问题
- 不 perfectionism——不 ship 完美的；ship 在改进路上
- 跨团队合作（policy / legal / ethics）
