## Q1：你怎么理解 Agent Harness 产品经理这个岗位？

**考察意图**：测你对 JD 的理解深度，以及是不是临时投机。

**答题骨架**：
1. 一句话定义岗位（**桥梁 + 翻译 + 自带动手**）
2. 三个核心职责的本质（不是 JD 抄背）
3. 你的优势 / 你的差距 / 你想用 90 天证明什么

**示范回答**：

> "这个岗位本质上是 **'连接研究员、用户、工程师'的产品翻译官**，但和传统 PM 不同的是——
>
> **第一，它不是把需求 spec 化扔给工程**——因为 Harness 的需求很多来自模型行为观察，需要 PM 自己能写 prompt、跑 demo、读 trajectory；
>
> **第二，它的指标体系比传统产品复杂一层**——'更多场景帮到更多人'要拆成 task success × user retention × scenario coverage 三轴，并且要避免 Goodhart's Law（比如盯 step 数会让 agent 偷懒）；
>
> **第三，它对'反馈飞轮'的设计权重远高于功能本身**——用户 trajectory 是模型训练的弹药库，PM 设计的是飞轮，不是 feature。
>
> 我希望加入的话，前 90 天先做三件事：(1) 用 DeepSeek 内部任务 dogfooding，定一版可信的 eval set；(2) 把社区前 100 个 bug report 自动聚类成产品反馈看板；(3) 起草一版 'Model ↔ Harness 接口契约'，和研究员对齐。"

**追问 + 应对**：
- ❓"你前面提到 dogfooding，怎么推研究员愿意用？" → 答案：用真正能省他们时间的场景切入（比如阅读 PR、写 release note），不要一上来推他们写代码（他们用 vim/emacs）
- ❓"你说的 trajectory 飞轮，合规怎么做？" → 答案：opt-in + 脱敏 + 分级（公开 issue 全收 / 内部 trace 走合规审查 / 用户私人代码绝不收）

**示范回答**（请按你的真实情况改写）：

> "最近一周我主要在用 **Claude Code 和 Cursor**，分别做不同的事——

## Q3：怎么定义"Agent 是否真的在更多场景下更深入地帮助到更多的人"？

**考察意图**：JD 职责 2 的直接挑战——这道题答得好不好基本定生死。

**答题骨架**：
1. 拆关键词：**更多场景（广度）× 更深入（深度）× 更多人（规模）**
2. 每个维度给 1 个北极星 + 2~3 个二级指标
3. 引入反向指标 + 防 Goodhart trick
4. 说出"我会怎么落地"

**示范回答**：

> "我会把它拆成三轴指标。
>
> **轴一·广度（场景）**：北极星 = **Active Scenario Coverage**——按用户任务的 intent 聚类（用 LLM 自动聚类成 30~50 个 bucket），过去 28 天 ≥ N 次活跃使用的 bucket 数。二级指标包括：new scenario unlock rate、scenario abandonment rate（某 bucket 用户 30 天后流失率）。
>
> **轴二·深度（任务完成质量）**：北极星 = **Autonomous Task Resolution Rate**——用户提交任务到 agent 自主完成（无 user intervention）的占比。二级指标包括：mean intervention per task、self-fix rate、task duration vs human baseline。这条要小心 Goodhart——如果只盯 resolution rate，agent 会学会"挑容易的做"，所以必须分难度分层。
>
> **轴三·规模（用户数）**：不是 DAU，而是 **WAU by depth segment**——分浅度（每周 < 3 次 invoke）/ 中度 / 重度（每天有 1+ 长 session）。重度用户占比是 PMF 信号。
>
> **反向指标**：(1) Interrupt rate（用户按 Esc 打断）—— ≥ 一定阈值说明信任度低；(2) Hallucinated tool args rate；(3) Token-per-success（成本健康度）。
>
> **落地**：第一阶段（前 30 天）我会先建 trajectory 数据管道 + scenario 聚类；第二阶段（第 31~90 天）和研究员对齐 eval set，把这些指标转成 model 可优化的 reward；第三阶段（第 91~180 天）灰度 A/B + 多场景纵向追踪。"

**追问 + 应对**：
- ❓"按 LLM 聚类的 scenario buckets 怎么保证稳定？" → 用 hierarchical clustering + manual seed bucket + 月度审核
- ❓"如果 R&D 不愿意按你的 reward signal 训，怎么办？" → 答 Q7
- ❓"如果用户根本没那么多怎么办（冷启动）？" → 答 Q5

## Q4：如果让你设计 DeepSeek Harness 的 v0.1，3 个月内要做什么？

**考察意图**：考你能否在信息不全的情况下做产品 trade-off。

**答题骨架**：
1. 一句话产品定位
2. 3 个月里**只做** 5 件事 + **明确不做** 3 件事
3. 第 90 天的 PMF 判定标准

**示范回答**：

> "v0.1 我会把它定位成：**'DeepSeek 模型的 reference harness，定位先抓两类种子用户——一是 DeepSeek 内部研究员/工程师，二是关注国产模型的开发者社区'**。3 个月，**只做 5 件事**：
>
> 1. **CLI 形态先行**（不做 GUI）——因为 CLI 出活快、低交付成本、聚精英用户；
> 2. **40 个核心工具 minimal set**（参考 Claude Code 的 40 个，但优先 read/write/bash/grep/glob/edit/test 这 7 个高频）；
> 3. **DeepSeek 模型原生 tool_use 适配**（包括 KV cache breakpoint 优化）；
> 4. **打通最关键的 1 个生命周期 hook：PreCompact**（让用户能注入自己的 handoff 逻辑）；
> 5. **内部 dogfooding 看板** + **公开 trajectory eval set**（带 100 个内部任务，做成 leaderboard）。
>
> **不做的 3 件事**：(1) 不做 Multi-Agent / Subagent（v0.1 不要复杂度，先把单 agent 跑稳）；(2) 不做 GUI（留给 v0.3）；(3) 不做完整 Skills 平台（v0.1 用 FS 挂载，社区先跑）。
>
> **第 90 天 PMF 判定**：(a) 内部 ≥ 30 名研究员/工程师每周自然活跃；(b) 公开 leaderboard 上 DeepSeek+Harness 在自家 eval set 上能跑赢 Claude Code+Sonnet 4.5 至少 5 个 pp；(c) GitHub ≥ 5,000 star、≥ 50 external contributor。三条全中=继续投；中 2 条=调方向；中 ≤ 1=认真讨论 pivot。"

**追问 + 应对**：
- ❓"为什么先做 CLI 不做 GUI？" → "CLI 是开发者最低摩擦的入口；GUI 早做会偏离技术核心，等模型 + harness 稳了再做 GUI 也来得及。"
- ❓"你说 5 件事，但内部 dogfooding 不算事？" → "我把它单列成第 5 件，是因为它本质是数据 pipeline 工程，不是 feature。"

**示范回答**：

## Q6：DeepSeek Harness 怎么和 Claude Code 形成差异化？

**考察意图**：竞品分析 + 产品判断。

**示范回答**：

> "我看到 4 个非对称机会，按优先级排：
>
> 1. **国产模型生态**（机会最大）：Claude Code 锁死 Claude，DeepSeek Harness 可以做 'multi-model first'——不仅 DeepSeek 自家，还兼容 Qwen/GLM/Kimi/GPT-OSS。代价是工程复杂度，回报是中文/全球非英语开发者愿意把它当 daily driver。
> 2. **完全开源**：Claude Code 闭源，DeepSeek 把核心 Harness 开源（哪怕只是 Apache 2.0 的 reference 实现），社区贡献 + 信任度都能起来。
> 3. **训练飞轮**：Anthropic 拿不到中文开发者真实 trajectory，DeepSeek 可以——但要 opt-in + 严合规。
> 4. **桌面 GUI（中长期）**：CC 是终端，对国内非极客开发者门槛太高；做一个像 Cursor 但 deeply integrated 国产模型的 GUI 形态。
>
> **我不会做的差异化方向**：(a) 单纯卷 system prompt 长度——这是负面差异化；(b) 抄 CC 40 个工具一一对应——这是同质化竞争。"

## Q7：你要和研究员合作设计 reward signal，研究员说"用户反馈太脏没法用"，你怎么办？

**考察意图**：跨团队冲突 + 对训练流程的理解。

**示范回答**：

> "我会按三步走：
>
> **第一步：先承认他对的部分**——用户反馈天然 noisy，直接当 reward 确实会污染模型（reward hacking）。
>
> **第二步：把'脏数据'分层处理**：
> - L1 = **强信号**（任务 verifiable success / failure，比如 PR 是否 merge）→ 直接给 RL
> - L2 = **结构化弱信号**（用户 retention、按钮点击、修改了 agent 的输出多少）→ 走 DPO/IPO 数据
> - L3 = **自由文本反馈**（GitHub issue / Discord）→ 不直接进 RL，先用 LLM 聚类成 eval 子集，让研究员人工抽样验证
>
> **第三步：让他先试 L1**——给他一个完全 verifiable 的 eval set（比如 100 个 GitHub issue 修复任务，有 ground truth）。如果他在这上能跑出新 reward 信号，他会自己来要 L2/L3。
>
> 我学过的核心是：**别和研究员谈'用户反馈很重要'，要给他不会反对的数据切片**。"

**追问 + 应对**：
- ❓"如果他还是不动，怎么办？" → 升级一层：拉陈德里同步现状，让 leader 做权衡（这不是逃避，是把跨团队冲突放回 leader 决策层）

## Q8：怎么看 Subagent 和 Multi-Agent 在 Harness 里的角色？

**考察意图**：技术深度 + 产品判断的结合。

**示范回答**：

> "我把它们分两个用途——
>
> **Subagent**：解决'单 agent 上下文失焦'——长任务时 spawn worker 隔离上下文，只把 summary 汇报。**典型场景**：read 大量文件、跑完整 test suite。Claude Code 的 Task tool 是参考。
>
> **Multi-Agent**：解决'单 agent 自评偏正'——典型是 Anthropic 3 月的 Planner/Generator/Evaluator GAN-style 架构。
>
> **我对 DeepSeek v0.1 的建议**：**Subagent 是必备**（不然长任务跑不通），**Multi-Agent 是 v0.2+ 才做**。原因——Multi-Agent 加复杂度，调起来痛苦，而且对模型本身的'听话度'要求高（容易 agent A 不听 agent B 的话）。Anthropic 在 Opus 4.5 出来后才敢做三 agent，DeepSeek 自家模型在 instruction following 没到那个水平之前先别做。"

## Q10：你最看重 Agent 产品的哪 3 个体验细节？

**考察意图**：测你的"用户体验感"是不是真的。

**示范回答**：

> "我个人最看重——
>
> 1. **可中断性 (interruptibility)**——按 Esc 后多久真的停？很多 agent 假装停了但还在跑后台 tool。Claude Code 在 stream 中即时 abort 这点做得好。
>
> 2. **错误归因的透明度**——失败时给出'是 tool 错了 / 模型理解错了 / 权限拒绝 / 网络'四类清晰区分，而不是一个红色 stack。Cursor 在这方面比 CC 弱。
>
> 3. **'下一步是什么'的预期管理**——长任务时 agent 主动说出"我接下来要做 A、B、C"。OpenClaw 在 trajectory 可视化做得好，但成本是 UI 重。CC 用 todo 列表实现了简化版。
>
> 这三个细节都不是 feature，是**对'人和 agent 协作'这件事的尊重**——它们决定了用户能不能信任 agent 在背后跑 10 分钟。"

## Q11：你怎么用 DeepSeek Harness "维护用户社群"？

**考察意图**：JD 职责 5——社区运营能力。

**示范回答**：

> "我会按 **'信号源 → 提取 → 闭环'** 三步建社群运营 SOP：
>
> **信号源（5 个渠道）**：GitHub issues、Discord、小红书、Twitter / X、weekly user call。各渠道 SLA 不同（GH 24h、Discord 4h、小红书 12h）。
>
> **提取（自动化）**：每天定时跑一个 LLM 聚合 job——把所有渠道 24h 内的发言聚类成 'bug / feature request / praise / model behavior issue / docs gap' 五类，每类 top-5 摘要进 Slack #user-pulse 频道。
>
> **闭环**：每个被采纳的 feedback 都在 release note 里 @ 原贡献者；每月发一封"This Month in Harness"邮件给所有曾提过建议的人。
>
> **关键 KPI**：(a) median time-to-acknowledge < 4h；(b) 闭环率（提了建议的用户在 60 天内被告知结果）≥ 70%；(c) 月度自然增长率（不靠付费）。"
