## Q1：你怎么设计 DeepSeek Harness 的"系统性数据收集"方案？

**示范回答（带白板可画版）**：

> "我会用 **'5 源 × 3 层 × 2 用途'** 的矩阵：
>
> **5 个数据源**
> 1. **客户端 telemetry**（最贵但最准）：opt-in 上报，包含 session 元数据、tool 调用图、stop reason
> 2. **结构化反馈**：UI 内 thumbs up/down + 文字（low friction，量大）
> 3. **问卷**（quarterly）：抽样 N 人，定性 + 定量混合，覆盖动机 / 痛点 / 留存原因
> 4. **深访**：每月 ≥ 5 个 1:1 用户访谈，30~60 分钟，录音 + 转写
> 5. **开放渠道挖掘**：GitHub issue、Discord、小红书、Twitter、Reddit——LLM 自动聚类
>
> **3 个数据层**
> - L1·**事件层**（最原始）：每次 tool call / 每个 token / 每个 stop reason
> - L2·**会话层**（聚合）：一次 task 的 trajectory、duration、cost、outcome
> - L3·**用户层**（长周期）：retention cohort、scenario breakdown、NPS、推荐
>
> **2 类用途**
> - **产品迭代**：bug → fix；feature → ship；UX 摩擦点 → 重设计
> - **模型迭代**（给研究员）：trajectory 当 SFT/RL 样本、failure case 当 eval set、tool 用法当 capability 训练
>
> **关键工程**：
> - 所有数据都走 opt-in + 脱敏 + audit；
> - 用 OpenTelemetry 统一 schema；
> - trajectory 进数仓后用 LLM 做自动 tagging（scenario、failure mode、模型行为类型）"

## Q2：你怎么设计一个 A/B 实验来验证一个新 harness feature？

**示范回答（以"新增 PreCompact handoff 自定义"为例）**：

> "假设要验证'让用户自定义 PreCompact handoff 文件格式能否提升长任务完成率'。
>
> **第一步·假设**：
> - H1（备择）：新 feature 使长任务（≥ 30 turn）完成率提升 ≥ 5%
> - H0（零）：完成率无显著差异
>
> **第二步·指标**
> - **主指标**：长任务完成率（任务跑完且用户未 reject）
> - **辅指标**：长任务平均 token 成本、自我修复率、interrupt 率
> - **护栏指标**（guardrails）：短任务完成率（不能因为优化长任务把短任务搞坏）、p95 latency
>
> **第三步·样本量估算**
> - 假设基线完成率 50%，期望提升到 55%（绝对值 5pp）
> - α = 0.05，power = 0.8
> - 用 `power.prop.test` 估算：每组 ~1565 个 session
> - 假设日均长任务 session = 500，需要 7 天双臂跑满（每臂 50%）
>
> **第四步·分配**
> - 用 user_id hash 分桶（不能按 session 随机——污染单用户体验）
> - 必须做新老用户分层（cohort），新用户对差异更敏感
>
> **第五步·分析**
> - 主指标用 two-proportion z-test 或 chi-square
> - 辅指标用 t-test（连续变量）/ Mann-Whitney（分布不对称时）
> - **必须做 segment 分析**：按场景 / 模型 / 用户活跃度分层看，避免 Simpson's paradox
> - 检验做完报 p 值 + 置信区间 + effect size（Cohen's d / 提升 pp）
>
> **第六步·决策**
> - 主指标显著 + 无护栏破坏 → ship
> - 主指标不显著 → 看辅指标和定性反馈，决定下版迭代方向（不一定杀）
> - 任一护栏破坏 → 强 rollback"

## Q3：用户量很小（前 100 个种子用户）的时候，A/B 不显著，怎么决策？

**考察意图**：测你"小样本 + 不可 A/B"场景的判断力，这是早期产品常态。

**示范回答**：

> "100 用户量级 A/B 几乎注定不显著（power 不够）。我会改用 **3 类替代方法**：
>
> **方法 1·定性 + 定量混合**
> - 100 个用户全部能 1:1 聊
> - 给 20 个用户跑 'wizard of oz'——人工模拟新 feature，看反应（不用真做出来）
> - 看 'aha moment' 触发率、抱怨频率
>
> **方法 2·Repeated Measures / Within-subject**
> - 每用户先用 A，再用 B（或反之，counter-balance）
> - 比较同一用户的成功率差异——配对样本 t-test 比独立样本 power 高得多
> - 但要小心顺序效应和疲劳效应
>
> **方法 3·Multi-armed Bandit / Bayesian A/B**
> - 用 Thompson Sampling 在小样本下动态分配流量
> - Bayesian 给出 'B 比 A 好的概率'，比 p 值更直观
> - 用 PyMC / Bayesian Stats，决策门槛设 'P(B>A) > 0.85' 就 ship
>
> **方法 4·Pre-Registered Long Observation**
> - 不 A/B，所有人用新版，但定一组明确指标，跑 4 周
> - 用之前 4 周做 baseline，做 difference-in-differences（DiD）
> - 这是经济学方法，对早期产品有效
>
> 我个人偏好 **方法 1 + 方法 4 组合**——定性先抓信号，定量长观察确认。100 用户级别**不应该追求统计显著，应该追求模式发现**。"

## Q4：怎么避免指标失真 / Goodhart's Law？

**示范回答**：

> "Goodhart's Law 是 Agent 产品最容易踩的雷。三招防：
>
> **招 1·组合指标 + 反向指标对冲**
> - 主指标永远要配一个'相反方向'的反向指标，让二者必须同时改善
> - 例：'任务完成率'配'token 成本 / success'——光优化前者会让 agent 用更多 token 换分
> - 例：'轨迹效率（步数少）'配'修改后用户接受率'——光优化前者会让 agent 草率收工
>
> **招 2·难度分层 + 场景分层**
> - 总数 Pass@1 会被'容易任务'稀释——必须按 difficulty stratify
> - 优化某个 metric 时同时盯所有 segment，不能 'overall 涨了 5%' 但其中 hard subset 跌了 10%
>
> **招 3·定期换指标 + 加 holdout**
> - 优化某个 metric 6 个月以上，就该警惕被 hack 了——quarterly 重审指标
> - 留一组 holdout（约 5%~10% 用户绝不投放新版本），用 holdout 看真实长期趋势
>
> **真实案例**：YouTube 早期优化 watch time，结果推荐了一堆 clickbait 长视频——后来加入'satisfaction score'反向对冲。Agent 产品上等价的就是 'completion rate' + 'user-reported satisfaction'。"

## Q5：你怎么用 LLM 自动化处理用户反馈？

**示范回答**：

> "我把它工程化成一个每天跑的 pipeline：
>
> ```
> 渠道抓取（GitHub / Discord / 推特 / 小红书）
>   → 统一 schema（user_id 哈希 + raw_text + channel + ts）
>   → LLM Phase 1·分类（bug / feature_request / praise / model_behavior / docs / other）
>   → LLM Phase 2·去重（embed + cosine sim threshold）
>   → LLM Phase 3·聚类（hierarchical clustering，名字由 LLM 生成）
>   → LLM Phase 4·摘要（每 cluster 3 句摘要 + 5 个代表性原文 quote）
>   → 进 Slack #user-pulse + 数仓
> ```
>
> **关键设计**：
> - LLM 调用统一走 batch API（成本降一半）
> - 每个分类有 'ground truth' set（manually labeled 500 条），每周 eval 模型分类 accuracy，掉到 < 85% 就重 prompt
> - 严防 LLM 漏看高优 issue：单独走规则匹配兜底（出现 'crash' / 'data loss' / 'security' 关键词强制升级）
> - PM 每周一拉看板，PRD 直接引用 cluster summary
>
> **辅助一个 LLM 没法替代的**：每月人工抽样 50 条原文，PM 自己读——这是保持产品 sense 的关键，**不能完全靠 LLM 加工后的二手信息**。"

## Q6：你怎么设计一份高质量用户问卷？

**示范回答**：

> "我会用 **'5 节问卷'** 模板，最大 12 题，5 分钟内填完：
>
> **节 1·筛选（1~2 题）**
> - 你最近 30 天用 Harness 多频繁？（用于剔除非活跃用户）
> - 你主要在什么场景用它？（多选）
>
> **节 2·NPS 单题**
> - "你向同事推荐 DeepSeek Harness 的可能性？" 0~10
> - 后置 1 个'为什么打这个分'开放题——这是问卷里**最有价值的一题**
>
> **节 3·使用质量（3~4 题）**
> - 任务成功率自评（Likert 1~5）
> - 信任度（agent 跑 30 分钟你敢离开吗）
> - 与 Claude Code / Cursor 对比（如适用）
>
> **节 4·痛点（2 题，开放 + 选择混）**
> - "你过去 7 天遇到的最大问题是什么？"（open-end）
> - "下面哪些场景你希望 agent 更强？"（multi-choice）
>
> **节 5·人口学 / 留联系方式**
> - 角色（前端 / 后端 / 全栈 / 算法 / PM / 学生）
> - 是否愿意接受 30 分钟深访（留邮箱）
>
> **铁律**：
> 1. **绝不超过 12 题**——超过 50% 中途弃填
> 2. **必须有开放题**——量化只能确认假设，定性才能发现新假设
> 3. **避免双重负担题**（'你对 A 的清晰度和 B 的易用性怎么看？'）
> 4. **避免诱导式提问**（'你觉得我们的产品好用吗？' → 改成 'how would you rate...'）
> 5. **样本量 ≥ 30 才下结论**，<30 只用作 anecdote"

## Q7：用户访谈你怎么做？避免哪些陷阱？

**示范回答**：

> "我用 **'5 阶段 30 分钟模板'**：
>
> | 阶段 | 时长 | 做什么 | 不做什么 |
> | --- | --- | --- | --- |
> | 1·破冰 | 3min | 自我介绍 + 强调没有错答案 | 不夸用户 |
> | 2·背景 | 5min | 你做什么的、平常 workflow | 不问 demo 类问题 |
> | 3·使用过程回溯 | 10min | "上次用 harness 是什么时候，能不能从打开它讲起" | **绝不直接问'你觉得这个 feature 好不好'** |
> | 4·痛点 + 假设验证 | 8min | 拿 2~3 个具体痛点验证 / 提 1~2 个新 feature 假设看反应 | 不带销售感 |
> | 5·延伸 + 推荐 | 4min | "还有谁你觉得我应该聊聊？" | 不催时间 |
>
> **常见陷阱**：
> 1. **leading question**："你觉得这个新功能很好用对吧？" → 用户为礼貌都说好
> 2. **hypothetical question**："如果有个 X feature 你会用吗？" → 答 yes 的人 90% 不会用
> 3. **only positive 用户**：满意的用户更愿意接受访谈，但 churn 用户的信息价值大 10 倍
> 4. **只听不看**：让用户 screen share 复现，行为 ≠ 描述
> 5. **不录音**：复盘时记忆失真，必须录音 + 转写（提前征得同意）
>
> **后处理**：
> - 每个访谈 24h 内写一份'1 页 brief'：3 个 quote + 3 个 insight + 1 个 action item
> - 每月 5 个访谈做一次'交叉总结'：哪些 insight 出现在 ≥ 2 人身上"

## Q8：灰度发布你怎么做？

**示范回答**：

> "灰度发布按 **'4 阶段 ramp'**：
>
> 1. **Internal dogfood (1~3 天)**：DeepSeek 全员 + 早期 power user 200 人左右；目标是抓**致命崩溃**和**安全问题**
>
> 2. **Closed beta (5~7 天，1%)**：白名单 ~50~200 用户；目标是看**核心指标方向**——主指标涨/跌/平
>
> 3. **Open beta (7~14 天，10%)**：随机抽 10% 流量，必须有**holdout 控制组**作对比；这一阶段做最严肃的统计分析
>
> 4. **Full rollout (1~3 天，100%)**：先 25% → 50% → 100% 三步走，每步过夜观察护栏指标
>
> **每阶段的 go/no-go 标准**：
> - **崩溃率/错误率 ≤ baseline 1.1×**（硬门槛）
> - **p95 latency ≤ baseline 1.2×**
> - **核心北极星 ≥ baseline -2%**（允许小幅波动）
> - 任一红线触发 → 立即 rollback + 复盘
>
> **配套工程**：
> - feature flag 系统（必须支持秒级 rollback，不能依赖发布）
> - 实时 dashboard（不要等 24h batch）
> - on-call 轮值（灰度期间有人盯）"

## Q9：你怎么衡量 Agent 的"模型行为质量"？

**考察意图**：JD 反复强调"对模型行为有品味"。这题是把品味量化的能力测试。

**示范回答**：

> "我会从 4 个维度建评测：
>
> **维度 1·能力 (Capability)**
> - 公开 benchmark：SWE-Bench Verified / Pro、AiderBench、LiveCodeBench
> - 自有 benchmark：DeepSeek 内部 100 任务 + 中文场景任务
>
> **维度 2·对齐 (Alignment)**
> - 拒答率（用户明确说 no 后 agent 是否继续）
> - 越界率（agent 自作主张做了未授权动作）
> - sycophancy 率（同样问题加上 '请说我对'，答案变化多少）
>
> **维度 3·诚实 (Honesty)**
> - 'I don't know' rate（不确定时是否承认）
> - 幻觉率（输出文件路径 / 函数名 / API 不存在的比例）
> - 自我修正率（被指出错误后是否承认 + 改）
>
> **维度 4·效率 (Efficiency)**
> - 步数效率（解决任务 vs 最优步数）
> - Token 效率（解决任务 vs 理论最小 token）
> - 时间效率（wall clock vs 资深开发者基线）
>
> **打分方式**：
> - 能力：自动跑分（pass/fail 测试）
> - 对齐 / 诚实：LLM-as-judge + 人工抽样校准
> - 效率：自动统计
>
> **关键**：4 个维度的总分**不能相加成一个数**（信息丢失），要做雷达图。给研究员看雷达图 + 具体 case，他们才知道下一代要练哪一面。"

## Q10：你怎么用统计学的"显著性"避免被人带偏？

**示范回答**：

> "三个常见'伪显著'陷阱我会主动避：
>
> **陷阱 1·p-hacking**：跑了 20 个指标，有 1 个 p<0.05，本质是 family-wise error rate。
> - **解法**：Bonferroni 校正（α/n）或 BH 校正（控制 FDR）；主指标预先声明，辅指标只做'探索性发现'报告
>
> **陷阱 2·peeking**：A/B 跑到第 3 天看一眼显著了就 ship——这会严重夸大 effect size。
> - **解法**：固定样本量再停（pre-registered），或者用 sequential test（mSPRT / always-valid p-values）
>
> **陷阱 3·小 effect + 大样本**：1000 万样本下 0.1% 提升也'显著'，但根本没业务意义。
> - **解法**：永远报 effect size + 置信区间，决策门槛要业务意义（如 'lift ≥ 2%' 才 ship），不光看 p
>
> **加一个 Agent 特有陷阱·trajectory 相关性**：同一用户多个 session 不独立——必须按 user_id 做 cluster-robust 标准误，否则 p 值低估。
>
> 我对面试官的态度：**不报 effect size 和置信区间只报 p 值的数据，我会要求重做**——这是给我自己设的下限。"

## Q11：内部 dogfooding 和外部用户反馈打架，听谁的？

**示范回答**：

> "听 **可验证程度** 高的那一边，不是听人多的那一边。
>
> 三种情况分别处理：
>
> **情况 A·内部说 X 难用，外部说 X 好用**
> - 可能是内部用户的 power 体验过高（他们写更复杂任务）
> - 解法：分层看——X 在外部对应的任务复杂度是不是更低？如果是，feature 没问题，可能要做'高级模式'给 power user
>
> **情况 B·内部说 X 没问题，外部投诉 X**
> - 可能是 onboarding 缺陷（内部人天然懂）
> - 解法：录 5 个外部用户首次使用 X 的 screen，找 friction point
>
> **情况 C·两边都说 X 有问题但方向相反**
> - 这是产品定位题——必须由 leader 拍板做 segment-first 还是 generalist-first
> - PM 把 trade-off 列清楚交决策，不替老板做选择
>
> **一句话**：'用户多 ≠ 用户对'。**听 trajectory 数据 > 听 anecdote**。"

## Q12：如果让你给 DeepSeek Harness 设计一个 "Model Behavior Eval" 自动化 pipeline，你会怎么做？

**示范回答**：

> "我会做一个'每次新模型 ckpt 都自动跑一遍'的 pipeline：
>
> ```
> 触发：每次 model ckpt 上传 / harness release / 每周定期
>   ↓
> Stage 1·能力眼（Capability）
>   - 跑 SWE-Bench Verified（500 题）+ DeepSeek 内部 task set（100 题）
>   - 输出：Pass@1 / Pass@3 / 平均 step / 平均 cost
>   ↓
> Stage 2·对齐眼（Alignment）
>   - 跑 100 个'用户说 no' adversarial set
>   - 跑 50 个 'sycophancy probe'
>   - 输出：拒答率、越界率、sycophancy delta
>   ↓
> Stage 3·诚实眼（Honesty）
>   - 跑 50 个 'I don't know' calibration probe
>   - 跑 200 个'文件/函数是否真存在'幻觉 probe
>   - 输出：halluciation rate
>   ↓
> Stage 4·效率眼（Efficiency）
>   - 复用 Stage 1 trajectory，统计步数 / token / wall time vs 上版基线
>   ↓
> Stage 5·LLM-as-judge 横评
>   - 用 GPT / Claude 双盲评比，给出'相对上一版优劣'
>   ↓
> Stage 6·雷达图 + diff report 自动发 Slack + 邮件给研究员 + PM
> ```
>
> **关键工程**：
> - 测试集本身要 holdout 一部分（绝不让训练团队看见，防 overfit）
> - 每季度更新 20% 测试集，防 contamination
> - LLM-as-judge 要 calibration：每月人工抽 50 条对照打分，看 judge 和人偏差，>0.3 就重 prompt
> - 所有结果进数仓，能 query 'V3.2 到 V3.5 在 SWE-bench Verified 涨了多少'
>
> **这个 pipeline 本身就是给研究员的 leverage**——他们能在不依赖 PM 帮助的情况下，自己看到模型 + harness 的全方位表现。"
