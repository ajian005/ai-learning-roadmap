## 二、全栈特化高频题

### Q1·你最自豪做的一个"被同事高频用的工具"是什么？

**示范回答**（按真实情况改）：

> "我在 [前司] 做过一个内部 trajectory viewer——背景：算法同事经常需要看新模型 inference 的具体输出，但 raw JSON 在 1000 步的 trace 里完全没法看。
>
> **Action**：
> 1. 1 周做了 V1：React + d3 timeline + 按 role / tool filter；后端 FastAPI 直接拉日志文件
> 2. 第 2 周让 3 个算法同事 try + 收反馈：他们要 keyboard shortcut + URL state + diff
> 3. 第 3 周加：keyboard nav、URL state、diff view
> 4. 第 4 周 dogfood：自己跟 5 个算法每天用 1 小时观察 workflow
> 5. 第 5~8 周持续迭代：加 LLM-as-judge score 标注、按 fail mode 自动聚类
>
> **Result**：
> - 6 周后 25 个算法 + 研究员每周用 ~3 小时
> - 算法 debug 时间 reportedly 减少 30~50%
> - 后来变成团队 standard tool
>
> **学到的**：内部工具的 ROI 不是"功能多"是"workflow 真的省"——必须每天自己用 + 真坐在用户旁边看。"

**追问 + 应对**：
- "如何收集反馈" → 1:1 shadow + Slack channel + 每月用户调研
- "用户不用你的工具怎么办" → 找 1 个 power user 做 ambassador；让他每周 demo 给团队
- "竞品 / 替代品" → 我评估过 W&B / LangSmith / LangFuse，列出适用场景差异，选自研因为 [理由]

### Q2·你和算法/研究员合作过吗？他们让你抓狂的瞬间？

**示范回答**：

> "和算法合作最大的反差是 **'优先级感'** 不同——
>
> 算法的优先级是"我现在 ckpt 跑出来，1 小时内要看 trajectory"。
> 我的优先级可能是"先把这个新 feature merge 了"。
>
> **抓狂瞬间**：有一次算法 8 点找我说"看不到我刚上传的 trajectory"——我刚开始没意识到这有多急，回了一句"我下午看"——结果是他要赶 9 点 leader review 的会，他 8:45 等不到给 leader 道歉了。
>
> 后来我学到 2 件事：
> 1. **算法找我十之八九是真的紧急**——不能用普通工程师 ticket queue 心态对待
> 2. **'我下午看'是 anti-pattern**——应该是 '我 10 分钟内告诉你 ETA'，让对方能 plan
>
> 现在我有个原则：**算法发我消息 5 分钟内必须有 first response**——不一定解决，但要 acknowledge + 给 ETA。"

**追问**：
- "和算法理论分歧（比如格式 / metric 设计）怎么处理" → 先承认对方专业（算法 by training 思考更严）+ 用证据说话 + 不在 group chat 撕，1:1 解决
- "算法用了你的工具但抱怨慢/卡，怎么办" → 1) 立刻问 reproduce steps；2) 自己 profile（不是问对方"具体多慢"）；3) 给一个 ETA + 临时绕路

### Q4·你怎么看"被骂"——内部工具被用户抱怨时怎么处理？

**示范回答**：

> "我有过几次被骂的经验。学到的：
>
> 1. **当场不辩护**：用户抱怨往往不是 'logic 上对'，是 frustration——先 'I hear you'，再问具体场景
> 2. **快速给一个 fix or workaround**：能 1 小时 fix 的当场 fix；不能 fix 的给个临时绕路 + ETA
> 3. **写到 retro / 学习**：抱怨往往揭示 product gap——记下来作为下一个 iteration 的 input
> 4. **持续 follow-up**：fix 完主动 ping 那个用户，请他验证；让他知道他的反馈被 take seriously
> 5. **不 take it personally**：内部工具的批评是免费的产品研究，比 silence 强 100×
>
> 反例：以前我有同事被骂后给用户 push back 'you should read the docs'——结果用户再也不用他的工具，悄悄自己造了个轮子。**用户走了不是用户的问题，是产品的问题。**"

### Q5·CTF / 安全背景怎么用在 Agent infra？

**示范回答**（如果你有 CTF 经验）：

> "CTF 训练给我两个直接受用的能力——
>
> **1. 红队思维**：看一段 tool 实现 30 秒能想到 5 个 attack vector。
> 例如：bash tool 接受 string 直接 exec，我立刻想：
>   - 命令注入（cmd injection）
>   - 路径遍历（path traversal）
>   - 资源耗尽（fork bomb）
>   - sandbox 逃逸
>   - 网络外泄（SSRF）
>
> 这种嗅觉是 senior security eng 训练出来的，写 agent infra 极有用——LLM 输出的代码本质上是 untrusted input。
>
> **2. 防御纵深思维**：单一防线必然漏；做安全要叠多层。Agent sandbox 我会做：
>   - L1：input filter（block 已知 dangerous patterns）
>   - L2：AST validation（解析 bash AST 看危险节点）
>   - L3：OS sandbox（seccomp / landlock）
>   - L4：container limit（resource quota / network policy）
>   - L5：audit log（all tool calls 记录）
>
> 一层挂还有四层兜。这就是 CTF 教会我的。"

**如果你没打过 CTF**：
> "我没正式打过 CTF，但平时关注安全：读 OWASP / 跟 LLM jailbreak research / 自己复刻过几个 prompt injection。
> 我相信 Agent infra 这个岗位的安全意识比 CTF 证书更重要——能力比标签更值钱。"

### Q8·一个真实的 Production Incident 故事

**示范回答**（按真实改写）：

> **Situation**：上线一个新 trajectory upload endpoint 后 30 分钟，PagerDuty 报 503。
>
> **Action**：
> 1. 看 alert：error rate 飙到 30%，集中在 upload endpoint
> 2. Rollback：先把这个版本 rollback 到 N-1（5 分钟搞定）
> 3. 看 logs：发现是 trajectory_steps 表的 unique constraint 在新代码下被 violate
> 4. Root cause：我的新代码 'INSERT OR UPDATE' 写错成 'INSERT'，重复上传报错
> 5. Fix：补正确的 ON CONFLICT；加 idempotent test case
> 6. Re-deploy：staging 跑 24 小时无异常后上 prod
>
> **Result**：30 分钟内服务恢复；事后写 incident report 5 条改进 action：
> 1. unique constraint 影响的 endpoint 必须 idempotent test
> 2. canary deploy 至少 30 min 而不是 5 min
> 3. integration test 加 "重复上传同 id" 用例
> 4. on-call playbook 加 "trajectory api 503" 章节
> 5. metric "upload_dupe_count" 上报
>
> **学到的**：production 错误 80% 是 happy-path test 不覆盖的 edge case；CI 必须主动想 anti-cases。"

### Q10·为什么 DeepSeek 全栈岗，而不是字节 / 腾讯 / 阿里同类岗？

**示范回答**：

> "三个具体理由——
>
> 1. **DeepSeek 是中国少有的'研究 first' 工程文化**——做内部工具的工程师不是被 PM 安排做需求，是真正成为研究员的 collaborator
>
> 2. **Agent / Harness 是 frontier 领域**——比传统 ML infra（如 Spark / Flink）更新、更没有 best practice、更 high leverage——我做的工具会被 50 个算法每天用，他们的 ckpt 会影响千万终端用户
>
> 3. **小团队 + 长期主义**：字节腾讯一个组 100+ 人，slot 都填满；DeepSeek 这个团队 < 50 人，每个工程师 own 一大块，长期来看 senior 速度更快
>
> 我看 5 年后的 DeepSeek 是 中国版 Anthropic——能造 frontier model + frontier harness 的少数公司之一。"

## 三、和不同同事打交道的应对清单

| 同事类型 | 沟通要点 |
| --- | --- |
| **算法 / 研究员** | 用 paper / 数学语言；先给 mental model 推测，再请 verify；ETA 必须明确 |
| **SRE / Infra eng** | 用 metrics / SLO 语言；先列影响半径，再讨论 root cause |
| **PM** | 用 user story / 数据语言；不深入技术细节，给替代方案的 cost 量化 |
| **设计师** | 用屏幕共享 + 文字描述；尊重审美，但 push back 不合理性能 |
| **leader** | 一句话 status；问题先 surface 不到山尽水尽；ask for direction 不 ask for permission |
| **外部社区 / open source** | 用英文 + 礼貌 + 提供 reproducible context |
