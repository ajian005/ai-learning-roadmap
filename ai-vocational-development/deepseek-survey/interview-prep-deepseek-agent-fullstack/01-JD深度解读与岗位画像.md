# 01 · JD 深度解读与岗位画像（Agent 全栈版）

> 目标：把"Agent 全栈开发工程师"这份 JD 从招聘话术还原成"真实团队工程需求"，并和已有两个 DeepSeek 岗位（PM / Harness Engineer）做差异对照——确保你投准、面准、答准。

## 一、JD 全文（DeepSeek Agent 全栈开发工程师）

> **职位描述**
> 1. 负责将业界领先的 Agent 开发工具集成到内部强化学习（RL）基础设施中，打通外部工具与内部训练体系的链路；
> 2. 负责 Agent 容器服务的日常维护，包括框架版本同步、依赖管理、环境稳定性保障等运维工作；
> 3. 搭建并完善 Agent 评测平台，开发轨迹查看、调试分析等可视化工具，提升模型迭代效率；
> 4. 维护和优化内部 Agent 集成框架，支持便捷对接 RL 训练框架及外部 Agent Scaffold（如 LangChain、OpenAI Agents SDK 等），降低研发接入成本；
> 5. 与算法团队紧密协作，参与 Agent 能力的工程化落地，推动 Agent 系统的可靠性与可扩展性提升。
>
> **职位要求**
> 1. 具备前端开发经验，熟练掌握 JavaScript/TypeScript，能独立完成前端功能开发；
> 2. 熟悉 Python、Rust，具备扎实的编程功底和工程能力；
> 3. 熟悉 Docker、Kubernetes 等容器技术，有容器化服务部署与运维经验；
> 4. 熟悉 Claude Code、Cursor、Copilot 等 AI 编程工具的使用与原理，有相关二次开发经验者优先；
> 5. 具备良好的代码规范意识，熟悉 Git 工作流、CI/CD、代码审查等工程实践。
>
> **加分项**
> 1. 有 Agent 评测平台、轨迹回放或可视化工具的开发经验；
> 2. 对大语言模型（LLM）有系统性认知，理解 Context Window、Token 机制等核心概念；
> 3. 熟悉 Agent 交互协议与规范，包括 MCP、Tool Use、Function Calling 等；
> 4. 深度使用 Claude Code、Cursor 等 AI 编程工具，具备高质量 Prompt 编写能力。
> 5. 参与过 CTF 竞赛。

## 四、主要职责逐句拆解

### 职责 1：负责将业界领先的 Agent 开发工具集成到内部 RL 基础设施中

| 维度 | 解读 |
| --- | --- |
| **真实活儿** | 写 LangChain Agent → 内部 trajectory schema 的 adapter；写 OpenAI Agents SDK 的 wrapper 让它和 RL trainer 数据格式对齐 |

| 维度 | 解读 |

## 六、JD 里没写但一定会考的"暗线"

### 暗线 1：你能不能"造工具给算法"
JD 5 条职责都围绕"造内部工具"——这意味着面试官会问：
- "你为同事造过什么工具？被多少人用？"
- "你怎么收集内部用户反馈？"
- "工具用得不顺，被骂过怎么办？"

**应对**：准备一个"我造给团队/同事的工具被用得很爽"的 STAR 故事。

### 暗线 2：你懂不懂"研究员/算法"的工作节奏
你不能只懂工程——你得理解：
- 算法上传新 ckpt 后 1 小时内要看到 eval 结果
- 算法 debug 时要能从 reward 倒推到具体 trajectory
- 算法跑 100 个 ablation，需要 batch viewer 一次看完

**应对**：T-6 之前用任何工具复刻 weights&biases / langfuse / langsmith 一个核心 view，证明你 study 过这类产品。

### 暗线 3：你 OOM 时不慌
容器运维一定有 OOM / pod crash / image pull fail 这类问题。会问你"怎么 debug"。

**应对**：准备 1~2 个真实的 production incident 故事（OOM、依赖冲突、镜像 corrupt 等）。

### 暗线 4：CTF 不是噱头，是"安全/边界意识"
CTF 加分意味着团队在意 **沙箱逃逸 / RCE / 任意 tool 执行漏洞**。Agent 给 LLM 执行 bash 是高危场景——团队需要有安全嗅觉的人。

**应对**：哪怕你没打过 CTF，也要能聊：sandbox 设计 / privilege escalation 防范 / prompt injection 危害 / supply chain 攻击。

### 暗线 5：你是"内部 happy path" 还是 "用户视角 obsessive"
内部 tool 工程师最容易做的事是"做完就完事"。差的工程师做工具时把"算法用着难"当成正常，好的工程师会 dogfood + 持续 polish。

**应对**：故事里包含"我自己每天用 + 收集反馈 + 持续迭代" 的细节。
