## 一、Docker 必考 6 题

### Q1·写一个 production grade Python service Dockerfile

> 答案见 [03 速查] 第 3.2 节——多阶段 + uv + distroless + non-root。

**追问**：
- "为什么用 distroless 而不是 alpine" → distroless 没 musl libc 兼容性问题（numpy 等 binary wheel 在 alpine 经常装不上）；alpine 优势是更小，但生态坑多
- "为什么 builder 不直接是 runtime" → builder 里有 gcc / dev headers / pip cache 等，runtime 不需要——减体积 + 减攻击面
- "镜像还能更小吗" → distroless 已经 ~50MB；再小用 scratch + 静态 binary（Rust / Go），或 chainguard images

### Q2·镜像构建很慢，怎么优化？

```
1. 把 deps 安装和 src copy 分开（先 COPY pyproject.toml，install，再 COPY .）
   ↓ 改 source 后 deps layer 仍能 cache

2. 用 BuildKit 的 mount cache：
   --mount=type=cache,target=/root/.cache/pip
   ↓ pip 缓存跨 build 复用

3. uv / pnpm 替代 pip / npm
   ↓ 安装速度 5×

4. CI 用 docker buildx + registry cache：
   --cache-from=type=registry,ref=myimage:cache
   --cache-to=type=registry,ref=myimage:cache,mode=max
   ↓ CI runner 之间复用

5. 减 layer 数（合并 RUN）+ 删 apt cache
   apt-get update && apt-get install -y --no-install-recommends X \
   && rm -rf /var/lib/apt/lists/*

6. 多平台 build 用 docker buildx + QEMU
```

### Q3·Docker 容器跑起来发现 OOM Killed，怎么定位？

```
1. docker stats 看 memory usage curve
2. docker inspect <container> 看 OOMKilled: true
3. 镜像里 dmesg | grep -i kill 看具体进程
4. Python：tracemalloc 或 memory_profiler
5. 看是否大文件 load 进 memory 一次（应 streaming）
6. 看是否 cache 无界增长（如 lru_cache(maxsize=None)）
7. K8s 里：limits.memory 调大 + requests 设合理
8. Pod 之外：node memory 不够也会被 evict
```

### Q5·docker-compose 里多服务依赖启动顺序怎么保证

```yaml
services:
  api:
    build: ./api
    depends_on:
      db:
        condition: service_healthy  # 不是 service_started！
      redis:
        condition: service_started
  db:
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 3s
      retries: 5
      start_period: 5s
```

**强调**：`depends_on` 默认只保证启动顺序，不保证 ready；要用 healthcheck + `condition: service_healthy`。

### Q6·镜像安全扫描怎么做？

```
1. CI 集成 Trivy / Grype 扫 CVE：
   docker scan myimage:latest
   trivy image myimage:latest --severity HIGH,CRITICAL --exit-code 1

2. SBOM 生成（Software Bill of Materials）：
   syft myimage -o spdx-json > sbom.json

3. Sign 镜像：cosign sign（避免 supply chain attack）

4. base image 必须用官方 + sha digest pin：
   FROM python:3.12-slim@sha256:abc...

5. 定期 rebuild（pull latest patches）：
   weekly cron in CI

6. distroless / chainguard image：减小攻击面
```

### Q2·liveness / readiness / startup 三个 probe 的区别？什么时候用哪个？

> 答案见 [03 速查] 第 4.3 节。
>
> **关键 trade-off**：
> - liveness 失败 → kill pod。**别太严**（短抖动也 kill 会 cascading restart）
> - readiness 失败 → 从 Service endpoint 摘除。**这是优雅 backpressure 工具**
> - startup 失败 → 阻塞前两者直到 pass。**应用启动慢必须用**（如 load ML model）

### Q3·Pod 启动失败怎么 debug？

```
1. kubectl get pods -o wide  # 看状态（Pending / CrashLoopBackOff / ImagePullBackOff）
2. kubectl describe pod <name>  # 看 events 区
3. kubectl logs <name>           # logs；前一次 crash 用 --previous
4. 进入 pod：kubectl exec -it <name> -- /bin/sh
5. 临时 debug 用 kubectl debug：
   kubectl debug -it <pod> --image=busybox --target=<container>
6. ImagePullBackOff：检查 secret / private registry credentials
7. CrashLoopBackOff：app 启动 crash；看 logs；可能 env / config 问题
8. Pending：resource 不够 / nodeSelector 不匹配 / PVC pending
```

---

### Q4·Rolling update 怎么实现 zero downtime？

```
1. strategy: RollingUpdate
   maxUnavailable: 0    # 严格 0 旧 pod 被杀前新 pod 必须 ready
   maxSurge: 1

2. 新 pod 必须 readinessProbe pass 才接流量
   - probe 检查必须真反映应用 ready（不只是端口开着）
   - 应用关闭时 SIGTERM 后 graceful shutdown，让 ingoing requests done
   - terminationGracePeriodSeconds: 30

3. preStop hook 关 listener：
   lifecycle:
     preStop:
       exec:
         command: ["sleep", "5"]
   # 让 Service endpoint 先摘除该 pod 再 stop 应用

4. PodDisruptionBudget 保证 voluntary eviction 时仍有 N 个 ready：
   minAvailable: 1 / maxUnavailable: 1

5. 多 zone 副本：topologySpreadConstraints
```

### Q5·HPA / VPA / Cluster Autoscaler 区别？

| | HPA | VPA | Cluster Autoscaler |
| --- | --- | --- | --- |
| 维度 | 横向（replica 数） | 纵向（pod resource） | 节点数 |
| 触发 | CPU / memory / custom metric | usage 推荐值 | 没节点放 pending pod |
| 风险 | 太多副本占节点 | restart pod 才生效 | 节点冷启动 ~min |
| 适合 | 普通 web service | 难估资源的 job | 大集群 |

> **常组合**：HPA + Cluster Autoscaler 是最常见组合；VPA "recommendation only" 模式当成估算 tool 跑。

### Q6·Helm 和 Kustomize 怎么选？

| | Helm | Kustomize |
| --- | --- | --- |
| 模板机制 | Go template | overlay / patch |
| 学习曲线 | 中（template 调试痛） | 低 |
| 分发 | chart repo / OCI | git fork |
| 多 env | values-{env}.yaml | overlays/{env} |
| 适合 | 第三方分发 / 模板复杂 | 内部多 env / 简单 patch |

> **常用**：内部应用用 Kustomize；分发给别人（如内部 SDK）用 Helm。
> ArgoCD 两者都支持。

### Q7·Secret 怎么管？

```
1. 不要 kubectl create secret 然后忘了
2. 用 external secret operator：
   - 后端用 Vault / AWS Secrets Manager / GCP Secret Manager
   - 集群内只引用，不存值
3. SealedSecret / SOPS：加密后 commit 进 git
4. CSI Secret Store Driver：以 volume 形式 mount
5. 避免环境变量传 secret（ps / proc 可见）→ 用文件
6. 自动轮转 + 应用支持 hot reload
```

### Q8·一个 K8s 集群规模 100 节点，pod 启动突然慢，怎么排查？

```
1. kubectl get nodes -o wide：节点 Ready 状态
2. kubectl top nodes：CPU / memory 抢占
3. 看 image pull：镜像太大 / registry 慢？用 local mirror 或 nydus
4. 网络：CNI plugin（Calico / Cilium）健康；DNS 解析慢？
5. etcd：list-watch 大量 pod 时 etcd 压力
6. kubelet 日志：journalctl -u kubelet 看 slow events
7. PV / PVC pending：storage class 容量 / Quota
8. CrashLoopBackOff 累积：清理失败的 pod
```

### Q2·CI 跑得慢怎么办？

```
1. 测试拆 unit / integration / e2e，PR 只跑 unit + 改动文件相关 integration
2. cache：
   - npm / pip / cargo cache
   - docker layer cache（buildx）
   - 测试 fixture
3. 并行：matrix + sharding（pytest-split / jest --shard）
4. fail fast：一个 job 挂 cancel 其他
5. 自托管 runner（如果硬件强 + 网络好）
6. nx / turbo affected：只 build/test 改动模块
7. flaky test 隔离 + retry 配额
```

### Q3·canary deployment 怎么实现？

```
方案 A·argo rollouts
  - 替换 Deployment 为 Rollout 资源
  - canary: { steps: [ {setWeight: 5}, {pause: {duration: 5m}}, ... ] }
  - 自动 analysis：Prometheus query 错误率
  - 失败自动 abort + rollback

方案 B·flagger + service mesh (Istio / Linkerd)
  - HTTP weight 切流
  - 自动 metric analysis

方案 C·istio VirtualService weighted routing 手动
  - 适合不上 rollouts 工具

DB migration 必须先做兼容：
  - V1 schema + V1 code 跑
  - V2 schema 上（兼容 V1 code）
  - V2 code 上
  - V1 schema 元素清理（一周后）
```

### Q4·CI 突然报"can't pull image"，怎么排查？

```
1. registry 是否 down（curl registry HEAD）
2. credentials 是否过期（rotate）
3. rate limit（DockerHub 100 pull/6h，需登录或自建 mirror）
4. 镜像 tag 是否还存在（CI 之前 image 被 GC）
5. CI runner 网络：DNS / proxy
6. image 太大 → 用 layer cache + shrink
7. 长期方案：每个团队 mirror registry + retry policy
```

### Q5·依赖版本如何治理？

```
1. lock file 必须 commit：
   - pip: uv.lock / pip-tools requirements.txt
   - npm/pnpm: pnpm-lock.yaml
   - cargo: Cargo.lock
   - docker: image digest sha

2. Dependabot / Renovate 自动 PR upgrades
3. SBOM 生成 + 扫描（trivy / grype）
4. 大升级走 canary 通道（先 staging）
5. LangChain 这类高变更率 pin minor 版本
6. 内部 framework 自己发包 → semver + changelog 严格
```

## 四、运维场景题：每天用得到

### 场景 1·凌晨 3 点 PagerDuty 报 API 503

```
1. 看 alert 详情：哪个 endpoint / region / latency 指标
2. 看 grafana：CPU / memory / 错误率 / latency 趋势
3. 看 K8s：pods 状态 / recent events / rolling update?
4. 看 deploy history：最近 deploy 时间 vs alert 时间
5. 如果可疑 deploy → rollback：
   kubectl rollout undo deployment/api
6. 看 logs：kubectl logs -l app=api --tail=200
7. 看上下游：DB / Redis / 第三方 API 健康
8. 短期 mitigation（rate limit 调严 / 缓存预热 / scale out）
9. 找到 root cause + 写 incident report + 改进 action
```

---

### 场景 2·依赖 LangChain，0.3 → 0.4 内部 API break

```
1. 评估 break scope：哪些 API 变了，哪些影响内部代码
2. 决策：
   a) 锁旧版（pin 0.3.x）+ 给 1 个月迁移窗
   b) 立刻迁移 + 升级
3. 选 a 时：内部 adapter 层吸收 break，业务代码不变
4. 选 b 时：写迁移脚本 + 全自动测试覆盖
5. 内部 adapter 层永远必要——隔离上游变化
6. CI 加 "next version preview" job 跑 nightly，提早发现
```

---

### 场景 3·算法说"我的 trajectory 上传了但平台看不到"

```
1. 让算法给 trajectory_id
2. DB 查：select from trajectories where id = ?
3. 没记录 → 上传链路问题：
   - 上传 logs（user / time）有吗
   - upload endpoint 是否 reached
   - validation 是否 reject
4. 有记录 → 渲染问题：
   - 前端 cache 过期？
   - filter 把它过滤掉了？
   - JS error 控制台
5. 修后告诉算法 + 留下 monitoring：
   - 加 upload success metric
   - 加 trajectory visible-after-upload latency 监控
```
