# 10 · 面试中临场策略与话术（工程师版）

> PM 版的临场策略（[../interview-prep-deepseek-harness-pm/10-面试中临场策略与话术.md](../interview-prep-deepseek-harness-pm/10-面试中临场策略与话术.md)）大部分通用——开场 60 秒、卡壳话术、视频细节、最后 30 秒都适用。
>
> **本文档只列工程师特化**：白板编码、think aloud、系统设计 4 阶段、代码失误挽回。

## 一、工程师开场 60 秒（区别 PM 版）

### 标准公式

**【1 句问候】+【1 句感谢】+【30 秒自我介绍（侧重工程背景）】+【1 句反向开场】**

### 示范

> "您好，[面试官] 老师，很高兴见到您。
>
> 谢谢您在团队刚组建的紧张期还给我这次机会。
>
> 我是 [姓名]，过去 [N] 年主要做 [简化方向]，技术栈以 [TS / Python / Rust ...] 为主。最近 [N] 个月把大部分时间花在 Agent 工程实践上——比如 [具体 1 件：用 X 复刻了 Anthropic Y 篇博客的 Z 方法]。
>
> **我面试前用 ~500 行实现了一个 minimal harness——包含 agent loop / tool dispatcher / sandbox / lifecycle hook，代码在 [GitHub URL]。如果今天聊到设计或源码，我们可以一起 review；如果先聊背景或编码也完全没问题。您希望从哪里开始？**"

### 关键差异（vs PM 版）

- 工程师强调 **"我有 repo"** 而不是 "我有提案 PDF"
- 工程师强调 **"代码 review"** 而不是 "产品 review"
- 工程师强调 **"技术栈"** 给一个具体的语言锚点

## 二、白板编码 think aloud 模板

### 核心：**全程出声**

> 沉默写代码 = 面试官评分时只能看你写出的代码。
> Think aloud = 面试官还能听到你的推理 / 决策 / debug 思路——多 50% 的评分维度。

### 7 阶段模板

```
1. Clarify (30s)
   - "让我先确认问题：输入是 X，输出是 Y，约束是 Z。"
   - "edge case 我想到 null / empty / 极大数据，需要处理哪些？"
   - "性能/正确性 谁优先？"

2. Approach (60s) 不写代码先想清楚
   - "我会用 [approach 1]，复杂度 O(N)；
      也可以 [approach 2]，复杂度 O(log N) 但实现复杂。
      我先做 approach 1，时间够再讲 approach 2。"

3. Skeleton (1-2 min) 主干结构
   - 函数签名 / 主循环 / 主分支，先把骨架打出来
   - "let me sketch the structure first, fill in details next"

4. Implementation (5-10 min) 主体编码
   - 边写边说"这一段在处理 X"
   - 遇到复杂逻辑停 10 秒，"let me think through this..."
   - 写完一段主动 sanity-check："这块对吗？让我用 [example] 走一遍"

5. Edge Cases (2-3 min) 主动处理
   - "现在加 null / empty 保护"
   - "如果 input is degenerate（如 size 1 / size 0）会怎么样？"

6. Test Walk-through (2 min)
   - 用一个具体 input 手动 trace 代码
   - "Input X → step 1 → step 2 → output Y ✓"

7. Improvements (1 min)
   - "如果时间够我会改进 [a / b / c]"
   - "Production 化我会加 [test / logging / error]"
```

### 卡壳时的 think aloud

```
❌ 沉默 30 秒
✅ "Hmm, let me think out loud——
    I'm trying to figure out how to handle X.
    Option 1 would be A but I'm worried about B.
    Option 2 would be C, which avoids B but adds D.
    I think I'll go with option 2; let me try."
```

**这种语言不止让面试官知道你在想什么，也让你自己 reason 更清楚。**

### 写错时的话术

```
❌ 默默删掉重写
✅ "等下——我刚写的这段有问题，let me trace through——
    yes, 当 input is X，这段会 [bug behavior]，
    我应该用 [fix] 处理。Let me update."
```

> **承认错 + 解释根因 + 改 > 默默修复**——前者是工程素养加分。

## 三、系统设计 4 阶段节奏

> 详细见 [06 文档]。这里是临场速查。

```
0:00 - 5:00   Clarify
  ┌────────────────────────────────────────────┐
  │ - 复述问题                                 │
  │ - 确认 scale / SLO / 用户 / 约束           │
  │ - 确认 out of scope                        │
  │ - "听起来对吗？开始进入设计 OK 吗？"        │
  └────────────────────────────────────────────┘

5:00 - 15:00  High-level Design
  ┌────────────────────────────────────────────┐
  │ - 画组件 box                               │
  │ - 标数据流方向                             │
  │ - 标关键 API                               │
  │ - "您觉得这个层级合理吗？继续深入？"        │
  └────────────────────────────────────────────┘

15:00 - 30:00 Deep Dive (选 2-3 组件)
  ┌────────────────────────────────────────────┐
  │ - 数据结构 / DB schema                     │
  │ - API 签名                                 │
  │ - Trade-off 显式                           │
  │ - "选哪个深挖？您来 drive。"                │
  └────────────────────────────────────────────┘

30:00 - 40:00 Scale + Failure
  ┌────────────────────────────────────────────┐
  │ - 10× / 100× 怎么扩                        │
  │ - 单点故障 mitigations                     │
  │ - 监控 / 告警                              │
  │ - Rollout / Canary                         │
  └────────────────────────────────────────────┘

40:00 - 45:00 Reflection
  ┌────────────────────────────────────────────┐
  │ - "如果再多时间我会深聊 [X]"                │
  │ - "我没覆盖到的弱点是 [Y]"                  │
  │ - 反问 1 个                                 │
  └────────────────────────────────────────────┘
```

### 系统设计常用话术

| 场景 | 话术 |
| --- | --- |
| 开场 clarify | "Before I start designing, let me clarify a few things..." |
| 标 trade-off | "选 X 这条路 trade-off 是 Y——优势是 A，代价是 B" |
| 引导深挖 | "我想深聊 [组件]，您觉得这是面试目标吗？" |
| 承认不熟 | "[X] 我没在 production 实现过，我的理解是 [Y]，请指正" |
| 主动 scale | "现在我们到 10× scale，瓶颈会出在 [Z]，我会用 [solution] 解" |
| 主动 failure | "如果 [组件] down 了，blast radius 是 [scope]，缓解方案是 [solution]" |
| 主动收尾 | "总结一下：我设计了 A / B / C 三层；最不确定的是 D；最值得继续聊的是 E。" |

## 四、代码失误挽回话术（工程师特化）

### 失误 1·写了几行发现 approach 错

```
❌ 假装没事继续
✅ "Stop——I think this approach won't scale to [edge case].
    Let me reconsider——I'll switch to [new approach] because [reason].
    Mind if I start over?"
```

**主动 switch 比死撑加分**：显示你识别 wrong path 的能力。

### 失误 2·跑 example trace 时发现 bug

```
✅ "Hmm, the trace gives [wrong] for this input.
    Let me trace where it goes wrong——
    Yes, at line N, I should be [fix] instead of [wrong].
    Let me update."
```

### 失误 3·用错 API / 不记得语法

```
✅ "I don't remember the exact API for X.
    In production I'd Cursor it; here I'll pseudo-code it as `foo()` and the implementation would be Y."
```

**老实说"不记得 + 我会怎么查" > 编一个错的**。

### 失误 4·复杂度算错

```
✅ "Wait, my earlier complexity claim was wrong.
    Actually because [reason], it's O(N log N), not O(N).
    Let me also see if there's a faster approach——"
```

### 失误 5·测试 case 跑错

```
✅ "The test output doesn't match what I expected.
    Two possibilities: 
    (a) my code is wrong, 
    (b) my expected output is wrong.
    Let me check (b) first because it's faster——"
```

> **debug 思路展示比代码正确更值钱**——这是 senior 工程师的标志。

## 五、技术深问的应对（研究员风格面试官）

### 场景 A·被问到不熟的 paper / 工具

```
✅ "I haven't read that specific paper.
    My adjacent knowledge is [related paper / concept].
    My guess is [paper] addresses [problem]. Is that close?"
```

**关联到你熟悉的相邻概念 + 推测 + 求 verify**。

### 场景 B·被问到非常细的数字 / 实现细节

```
✅ "I don't remember the exact number/detail.
    The order of magnitude I remember is [estimate].
    Mind sharing the precise value? I want to update my mental model."
```

### 场景 C·被质疑你的判断

```
✅ "You're raising a fair point.
    Let me reconsider——
    if [your concern] is correct, then my approach would [fail mode].
    Is there context I'm missing that makes [concern] particularly important here?"
```

**不辩护，邀请对方做老师**。

### 场景 D·被连环追问 3 个 why

```
第 1 个 why：直接答
第 2 个 why：答 + "这是个 deep question，我之前没想这么细"
第 3 个 why：**慢下来**——"Let me think for 30 seconds"
            然后老实说"我推到这层就到知识边界了，需要 [资料/实验] 才能再深"
```

**第 3 层不会就承认到第 3 层**。

## 六、Live Coding 的工具栈细节

如果是远程 live coding（共享屏幕 + IDE），注意：

| 项 | 准备 |
| --- | --- |
| **IDE 选择** | 用你最熟的（VSCode / Cursor / Vim / Emacs）；不要面试当天用没用过的 |
| **AI 辅助开关** | 一般会要求关 Cursor AI / Copilot——提前测试如何快速关 |
| **字号** | 调大（≥ 14pt），让面试官也看得清 |
| **语言** | 优先用你 daily 用的，Python / TS / Go 最稳；除非面试官特别要求 |
| **Linter / Formatter** | 关掉，避免面试中突然格式化代码打断思路 |
| **测试** | 准备一个 minimal test runner（`python -c` / `node -e`），不依赖 IDE |
| **网络** | 提前测试共享屏幕延迟 |

## 七、Take-home assignment 应对

如果给 take-home（24~48 小时一个 mini project）：

### 原则

1. **不要拖到 deadline 前 1 小时交**——交得早 + 质量好显示态度
2. **README 是 deliverable 的一半**——清晰的 setup / decisions / trade-off doc
3. **不要 over-engineer**——做 spec 要求的事 + 1-2 个"如果有更多时间会做"的 stub

### 我建议的 take-home 工作流（48 小时）

```
H0-H2    读 spec 3 遍，列 questions，给 reviewer 发 clarifying email
H2-H4    设计 + skeleton（结构、interface、data flow）
H4-H10   核心实现
H10-H14  testing
H14-H18  README + 文档
H18-H20  refactor + cleanup
H20-H22  submit + 1 封 short summary email
H22-H48  buffer，可能 2 轮 reviewer 反馈
```

### README 模板

```markdown
# [Project Name]

## What this is
[1 paragraph]

## Quick start
[exact commands to run]

## Design decisions
- Decision 1: [what + why + alternatives considered]
- Decision 2: ...

## Trade-offs
- [What I chose to NOT do + why]

## If I had more time
- Item 1
- Item 2

## Test approach
[how to verify]
```

## 八、临场"30 秒口袋话术"速查（工程师特化）

| # | 场景 | 话术 |
| --- | --- | --- |
| 1 | 想想再写 | "Let me think out loud for 30 seconds before coding." |
| 2 | 没听懂 | "I want to confirm—are you asking X or Y?" |
| 3 | 不会答 | "I haven't directly worked with this; my mental model is X. Please correct." |
| 4 | 被打断 | "Good catch—let me adjust." |
| 5 | 被质疑 | "Fair point—if [concern] is correct, my approach would [fail]. Mind sharing the context?" |
| 6 | 写错代码 | "Stop, this approach won't handle X. Let me switch to Y." |
| 7 | API 不熟 | "I don't remember exact API; I'll pseudo-code as foo() and explain intent." |
| 8 | 给方案 | "I'd do A, trade-off is B; if priority is C I'd switch to D." |
| 9 | 反送球 | "What's your team's take on this? I want to see if I have blind spots." |
| 10 | 收尾 | "To recap, my approach is X, key trade-off is Y, biggest unknown is Z." |

## 九、一句应试钥匙

> **工程师面试的核心不是"代码完美"，是"思考过程透明 + 失误可恢复 + trade-off 显式"。**
>
> 面试官看的不只是代码——是**未来 1 年和你 pair programming 的体验预演**。
>
> 让对方在 pair 完之后想"我希望和这个人写代码"——offer 就到了。
