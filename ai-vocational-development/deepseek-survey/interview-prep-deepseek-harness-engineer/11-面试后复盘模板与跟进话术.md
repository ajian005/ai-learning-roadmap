## 二、面试后 V0.2 跟进策略（工程师独有杀招）

> 这是工程师面试 follow-up 最有力的武器——24~48 小时内**push 一个 V0.2 commit + 给面试官 link**。

### 时间线

```
T+0 当天晚上    | 复盘表填完 + V0.2 改进列表
T+1 早晨        | 实施 V0.2（2~3 小时）：把面试聊到的 1~2 个 weakness 补上
T+1 中午前      | git push + 发感谢邮件 + 附 V0.2 commit URL
T+1 ~ T+3       | 等回复；不主动 follow-up
T+4 或之后      | 礼貌跟进
```

### V0.2 改进的"小而有信号"清单

不要 V0.2 改一切——挑 1~3 个**能精准回应面试讨论**的小改进。

| 面试聊的主题 | V0.2 可加的 1 件事 | 1 个 commit message 示例 |
| --- | --- | --- |
| 聊到长任务 / context anxiety | 加 `context-reset` 模式 + handoff schema | `feat: add context-reset with handoff artifact (#1)` |
| 聊到 sandbox | 加 `--sandbox=os` 用 OS-level / bubblewrap | `feat: optional OS-level sandbox via bubblewrap (#2)` |
| 聊到 MCP | 加一个 minimal MCP client | `feat: bare-bones MCP stdio client` |
| 聊到 cache | 加 cache breakpoint marker | `perf: cache_control on system prompt section` |
| 聊到 concurrency | 加 concurrency-safe partition | `feat: dispatch parallel-safe tools in batch` |
| 聊到 streaming | 加 SSE streaming | `feat: SSE streaming output` |
| 聊到 schema | 加 Zod validation + retry | `feat: schema validation with retry on error` |

**关键**：每个 commit message 短而清晰，让面试官 30 秒能看懂。

## 三、感谢邮件模板（工程师版）

### 模板·给工程师/研究员面试官

```
主题：Re: 今天面试 - 几个延伸思考 + V0.2 push

[姓名/老师好]，

谢谢您今天的时间，特别是您聊到的 [具体技术话题] 让我回家又想了一晚。

简短分享两个回家后想清楚的：
1. 关于您问的 [Coding 题或概念]：面试时我答 [答案]，回家后觉得更准确的角度是 [新观点]——主要因为 [理由]
2. 关于您聊到的 [Harness 设计点]：我去查了 [资料/source]，进一步想到 [insight]

刚才在 minimal-harness 加了一个 V0.2 commit 来反映今天的讨论：
🔗 [https://github.com/you/mini-harness/commit/abc123]

主要改：
- [改动 1，一行描述]
- [改动 2，一行描述]

期待保持联系。

[你的名字]
[GitHub / Twitter]
```

### 邮件 5 原则（工程师版）

1. **24 小时内发** —— 越快越好
2. **不超过 200 字** + **附 1 个 commit URL** —— 真实的 deliverable
3. **必须提到面试中的 1 个具体话题** —— 不是套话
4. **commit message 写得 production-ready** —— 不要 'wip' 或 'asdf'
5. **抄送 HR** —— 保持透明

## 五、被拒后的工程师特化转化

### 转化 1·求 technical feedback（重要）

```
谢谢告知。完全尊重决定。

如果方便，能不能请您（或团队）分享 1 个最 specific 的 technical feedback？
比如"system design 在 [X] 没讲清楚"、"coding 题 [Y] 写得不够好"、"对 [Z] 概念理解不够"。

这对我下一段成长意义大于 offer 本身。
```

> 工程师 feedback 比 PM feedback 更易拿到——技术问题更"objective"。

### 转化 2·建立技术对话

```
完全理解。我会继续关注 DeepSeek Harness 的进展。

我打算把面试中聊到的几个有意思的话题写一篇博客：
- 主题：[根据面试聊的内容选]
- 草稿大致 1500 字

写完发给您 review 可以吗？也欢迎您 disagree。
```

> 把"被拒"变"技术朋友"——长期价值远大于这次 offer。

## 七、心理建设：工程师的拒绝远多于 offer

| 拒绝原因 | 真实概率（工程师面试） |
| --- | --- |
| 编码 / 设计能力不够 | 30% |
| Bar raiser 的"对照组"更强 | 25% |
| 文化 / fit 不匹配 | 20% |
| 业务/团队短期变化（hiring freeze） | 15% |
| 太贵 / 太便宜 / 地域 | 10% |

> 70% 的拒绝**不是你能力差**——但 100% 的拒绝都能拿到可迁移经验。

> **接受 5+ 次拒绝是 Agent 工程师 senior 路径上的常态**——尤其是 Anthropic / OpenAI / DeepSeek 这种 talent-dense 团队。

---

## 八、一句应试钥匙（工程师版）

> **复盘的最高价值不是"为下次面赢"，是"看清自己作为 Agent Engineer 在 2026 年的真实段位"。**
>
> 段位看清了——下一步是补编码、补系统、补 paper 阅读、补开源贡献，每个方向都有具体动作可做。
>
> **面试是镜子，复盘是显影液——但 commit log 是你 1 年后回头能看到自己成长的实证。**
