### Q2：你怎么和研究员沟通技术问题？

**示范回答**（用故事 4）：

> **Situation**：之前在 [X 项目]，研究员 R 训出新一版 reward model，希望我把它接入 inference pipeline，但 R 给的 spec 文档只有 1 段话。
>
> **Task**：我作为 infra eng，要在 1 周内 ship integration，但 spec 不全。
>
> **Action**：
> 1. **不直接问'怎么写'**——先自己读 R 之前的 paper / 代码，列出'我推测的接口契约 5 条'，发给 R 让 R verify / correct，比让 R 从头讲省他 90% 时间
> 2. **用 R 听得懂的话提需求**：不说'我要文档'，说'我打算用 X 函数签名 Y 数据格式，你看可否'
> 3. **遇到性能问题时不甩锅**——自己先跑 profile，定位到 'tokenization 占 60% 时间'，给 R 看 flame graph 再讨论
> 4. **deliverable 是 PR 加 design doc**，让 R review，给 R 提 visibility
>
> **Result**：1 周 ship，R 后来主动找我做下一版。**学到一件事**：研究员的时间是稀缺资源——PM/Eng 找 R 之前必须先把 30% 的活做完，让 R 决策剩下的 70%，不是从 0 开始。

**追问 + 应对**：
- ❓"研究员说他的方法是 SOTA，但 benchmark 数据看起来不太对，怎么处理？" → 不质疑结论，先确认 setup ("能不能 share 你跑的 eval 命令")；自己复刻一遍；用证据说话
- ❓"研究员推一个工程上特别难实现的需求，怎么办？" → 不说不能，说"按你这个方案 cost 是 X 时间 Y 钱，按 alternative 是 X' Y'，你看哪个 trade-off 更合"

**JD 任职要求 3 核心**

> **例 1**：去年我从 Python/JS 背景，1 个月做出一个 Rust + Tokio 的 [项目]，过程中我 (a) 用 Cursor 跑 Rust by Example，(b) 每写一段先让 Claude 解释 lifetime / borrow checker 报错，(c) 找了 Rust async 的 3 篇核心 blog 啃完。**结果**：项目上线后跑了 6 个月零 panic。
>
> **例 2**：今年初我需要写一个 Bun-based CLI，之前没用过 Bun。用了 4 小时把 Bun runtime / bundle / FFI 撸明白，做了一个 [URL]。**关键**：我没等到全部学完才动手，是 'just-in-time learning'——遇到一个问题学一个 API。

> **例 3**：上周面试准备这个 minimal harness，我第一次用 SQLite + sqlx（Rust），用 Claude Code 配合 1 小时跑通基础 CRUD，2 小时实现 trajectory 持久化。

> **我的 workflow**：
> 1. 让 Claude 先生成项目结构 + Hello World，跑通

> 3. 每实现一个 feature 先 'show me an example' 再改造
> 4. 错误信息直接喂给 Claude 让它解释根因（不是修复方案，是根因——避免 'fix the symptom not the disease'）
>
> 这是 PM JD 里的 'vibe coding'，工程师 JD 里要求 '有质量保证'——多出 '质量' 二字意味着我不止跑得通，还要让代码可读 / 有测试 / 性能合理。"

### Q4：你为什么参与开源？你最骄傲的 contribution 是什么？

**示范回答**（按你真实情况改写）：

> "我参与开源主要为三件事——

### Q5：DeepSeek 工程师面试以"3 小时长面 + 高强度压力"著称——你怎么准备和应对？

**示范回答**：

> "我把它当成 'work session' 而不是 'interview' 来准备。
>
> **物理准备**：
> - 睡眠：前一天 23 点前睡，保证 7h+
> - 饮食：面试前 1 小时吃 8 分饱 + 1 杯黑咖啡
> - 中途补水：随手准备水
>
> **心理准备**：
> - 接受 3 小时不会一直 'on'——会有低谷期，那就慢下来 30 秒
> - 接受 connection lost——遇到完全不会的题不死撑，主动说 'I don't know but my best guess is...'
> - 压力面里，**面试官质疑你不是因为你错，是测你怎么 reason**——别 take it personally
>
> **具体技巧**：
> - 每题答完主动'切换信号'：'这题我答到这；下一题来吧'
> - 写代码不行时，主动切到 design level：'让我先讲思路，代码我们后面 fill in'
> - 喝水也是 reset 节奏——一题答完喝一口水，给自己 3 秒缓冲
>
> 我相信 DeepSeek 的 3 小时不是想 '难住我'，是想看我'在长时间认知负担下还稳不稳'——这正是 Harness 工程师每天的工作状态。"

### Q6：你看过最近的哪些 agent / LLM paper？说说你的看法

**示范回答**（按你真实读过的换；下面是参考）：

> "最近读了三篇印象比较深的——
>
> **1. SWE-Bench Pro (arXiv 2509.16941)**：长 horizon SWE 任务，多文件，pass@1 现在最强模型也 < 45%。这告诉我们 (a) 真实世界编码比 'fix one file' 复杂得多 (b) Harness 设计对长任务的杠杆远大于换模型。
>
> **2. Anthropic 'Harness design for long-running application development' (2026-03)**：GAN-style 三 Agent。最让我触动的是 'generator-evaluator separation'——单 agent 自评偏正，分开 agent 用独立 context + Playwright 真测，能突破质量上限。我自己复刻过 evaluator agent，确实能挑出 generator 看不到的 UI bug。
>
> **3. Voyager (2305.16291)**：在 Minecraft 里 build skill library，自动 curriculum 学习。它对 Harness 的启示是——**Skills 不应该是开发者手写的，应该是 agent 自己 distill 出来的**——这是 DeepSeek Harness 一个可能的差异化方向。
>
> 我有一个 reading queue 在 GitHub：[URL]，每篇都有我的笔记。"

**追问 + 应对**：
- ❓"你最不同意哪篇的观点？" → "[paper X] 认为 [观点]，但我实测 [实验] 发现 [反例]，可能是 [原因]——这是后续 research 方向" → 显示你有独立 judgment
- ❓"如果让你写一篇 paper，主题会是？" → 提一个具体 working title + 你的 hypothesis（即使粗糙），证明你有 research 品味

### Q7：你和算法/research 同事经常意见分歧？怎么处理？

**示范回答**：

> "我处理意见分歧的 3 步：
>
> **Step 1·先寻找 underlying concern**：
> 我学到 90% 的分歧不是表面观点的对错，是 underlying concern 没对齐。比如研究员说 'eval 必须 holdout'，工程师说 '我没那么多数据'——背后是 '研究员怕 overfit'，'工程师怕 underpowered'。看到 concern 后，混合方案（80% shared + 20% holdout）就出来了。
>
> **Step 2·用证据说话**：
> 不说'我觉得你这样不行'，说'我跑了 [实验]，看到 [数据]，所以我建议 [方案]'。和研究员对话尤其重要——他们 by training 不接受 anecdote。
>
> **Step 3·接受'this is their call'**：
> 模型架构决策最后是研究员 call，工程实现细节是我 call。明确边界后，超出我边界的我让步——但记下来观察长期效果，如果证明我的判断对，下次提案带着 data。
>
> 我从来不会把分歧带到团队公开冲突——1:1 解决，1:1 不行升级到 leader。"

### Q8：你怎么看 AI 取代工程师这件事？

**示范回答**：

> "我觉得 AI **取代了 60~70% 的传统编码工作**——'写 CRUD' / '套模板' / '调 API' 这种活，Cursor / Claude Code 已经做得比初级工程师快 + 准。
>
> **但取代不了的核心 30%**：
> 1. **System-level 决策**：技术选型 / 架构权衡 / 性能 vs 复杂度 trade-off
> 2. **跨学科桥梁**：研究员 ↔ 用户 ↔ infra 之间的语义翻译
> 3. **debug 'unknown unknowns'**：AI 还修不了它没见过的 bug
> 4. **审美与品味**：哪段代码该重写 / 哪个 abstraction 该 cut
> 5. **承担工程决策的政治成本**：say no 到 powerful stakeholder
>
> 我的方向不是'防 AI 取代'，是'**用 AI 把那 60% 自动化，把 100% 时间花在 30%**'。这就是 JD 写'熟练使用 AI Agent 工具进行开发'的核心——不是要你学新工具，是改变你工作的 leverage。
>
> Agent Harness 工程师这个角色尤其特别——**我们做的产品，就是在改变 AI 怎么帮其他工程师**。我自己每天用 Claude Code / Cursor 写 90% 代码，对 Harness 的设计有第一手感知。"

### Q10：你最大的弱点是什么？

**示范回答**（请按真实情况改）：

> "我比较诚实的弱点是 **'倾向把所有边界 case 在一开始就考虑完，导致 over-engineering'**。
>
> 这在 'production system' 里是优势，在 '0→1 prototype' 里是负担——一个 hackathon-size 的 demo 我会花 50% 时间想 'how it scales to 10K user'，但 demo 阶段根本不需要。
>
> 最近我在做反向训练——给自己定 'time-box first version'：第一版必须在 N 小时内 ship，不允许加 edge case。后面再迭代。这次 minimal harness 我就是这么做的——v0 简陋但 1 小时跑通；v1 才加 error handling / cache。
>
> 这件事我还会继续调，因为 0→1 团队对 'ship 速度 > 完美设计' 的容忍度极高。"

## 四、DeepSeek 工程师压力面 5 条心法

> 同 PM 版的 5 条心法（在 [../interview-prep-deepseek-harness-pm/07-...md](../interview-prep-deepseek-harness-pm/07-高频面试题与参考答案-行为与文化篇.md) 第四节）适用，工程师特化补充：

1. **写代码遇阻时主动'切换信号'**：白板憋 5 分钟没思路 → 主动说 'let me switch to discussing the approach first' → 防止"沉默死循环"
2. **代码错被指出**：'You are right, let me trace through again' → 当场 trace → 显示 debug 能力
3. **不会的概念**：'I haven't worked with X directly, my mental model is Y, please correct me' → 显示学习姿态
4. **质疑你的判断时**：先复述对方观点（'so you are saying Z would be better because...'），再分析 trade-off
5. **思路不收敛时**：'I am going to commit to approach A because of B; I know C is also valid but I'd rather depth than breadth in this time' → 显示决断

## 五、行为面"反模式"清单（工程师特化）

| 反模式 | 为什么扣分 | 替代 |
| --- | --- | --- |
| 写代码不出声 | 面试官不知道你在想什么 | think aloud 全程 |
| 不写测试 | 显示工程不严谨 | 主动说 "I'd add a test for X" |
| 直接抛异常 | 看不到 error handling 思考 | 显式 try/catch + 注释为什么 |
| 一气呵成完美代码 | 看起来背的 | 主动 refactor "let me clean this up" |
| 用 we 不用 I | 看不出贡献 | "团队做了 X，我个人负责 Y" |
| 报喜不报忧 | 不真实 | 主动说"if I redo I'd..." |
| 抱怨前司 | 你来了也会抱怨 | 抽象描述，不评人 |
| 不问问题 | 显示没思考 | 反问环节至少 ≥ 3 个 |

## 六、一句应试钥匙
